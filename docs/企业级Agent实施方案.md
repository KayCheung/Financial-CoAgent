# 企业级 Agent 平台分阶段实施方案

> **基准文档**：基于 DeerFlow 的企业级 Agent 平台完整方案 v1.5  
> **编制日期**：2026-04-08  
> **核心原则**：每阶段可独立交付成果 · 任务间低耦合/接口化依赖 · AI 辅助编码加速

---

## 总览：四阶段里程碑

| 阶段 | 周期 | 核心目标 | 可见成果 |
|------|------|----------|----------|
| **一期：基座闭环** | M1-M2（8周） | 跑通对话 + OCR 发票全链路 | 用户可正常对话、上传发票自动识别并生成凭证 |
| **二期：Harness 构建** | M3-M4（8周） | 安全护栏 + 记忆体系 + HITL | 工具拦截生效、审批流程可用、多轮上下文连贯 |
| **三期：业务深水区** | M5-M6（8周） | 脱敏网关 + MCP 问数 + 财务Agent | 财务报销自动闭环、审批卡片、PII 脱敏可用 |
| **四期：平台化运营** | M7-M10（16周） | Skill 广场 + 可观测 + 生产就绪 | 全链路监控、发布门禁、压测通过、全面投产 |

## DeerFlow 基座评审前提

本方案默认基于 DeerFlow 2.0 + LangGraph 1.0 的工作流底座。以下内容在进入各阶段实施前必须先确认：

- DeerFlow 在本项目中只承担工作流编排内核，不默认包含全部业务能力。
- `Tool`、`Skill`、`Agent` 的职责边界要统一，不能在不同任务里各自解释。
- `thread_id`、`trace_id`、`tenant_id` 等上下文字段要贯穿网关、引擎、工具、审计层。
- 状态持久化优先采用 Redis Checkpointer + PostgreSQL 审计，恢复和回放策略先定。
- 模型接入层默认只做统一网关与熔断，不在首期引入复杂 prompt 平台。
- 涉及跨租户、跨权限、跨审批的能力，默认必须经过网关和审计，不允许旁路调用。

## 第一阶段：基座闭环（M1-M2，8周）

### 阶段目标

**让用户看到可交互的成果**：能正常对话问答、能上传发票图片完成 OCR 识别并输出结构化结果。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 对话可用性 | 用户通过 Web/API 发送自然语言问题，Agent 在 10s 内返回有效回复 |
| OCR 发票识别 | 上传标准增值税发票图片，关键字段（发票号、金额、税号、日期）提取准确率 ≥ 90% |
| 路由命中率 | Semantic Router 在冷启动配置下，意图分类准确率 ≥ 70% |
| 系统稳定性 | 单用户连续对话 50 轮无崩溃，API 可用率 ≥ 99% |
| Token 成本可观测 | 每次对话的 Token 消耗可在日志中查看 |

### 所需中间件

| 中间件 | 版本 | 用途 | 部署要求 | 最低资源 |
|--------|------|------|----------|----------|
| PostgreSQL | 15+ | 业务数据、事件日志、审批记录 | 单实例（规格A），开启 WAL | 2C4G, 50GB SSD |
| Redis | 7.0+ | 会话状态、Token 计数、SR 缓存 | 单节点（规格A） | 1C2G, 10GB |
| Milvus | 2.3+ | Semantic Router 向量检索 | Standalone 模式 | 2C4G, 50GB SSD |
| Kong/APISIX | 3.6+/3.0+ | API 网关，限流/JWT/路由 | 单节点 | 1C2G |
| PaddleOCR | PP-Structure | 发票版面解析 | Docker/GPU 可选 | 2C4G（CPU）或 1GPU |
| Nacos | 2.3+ | 配置中心（热更新） | 单节点 | 1C2G |
| Python Runtime | 3.11+ | DeerFlow + LangGraph + LangChain | 容器化部署 | 4C8G × 2 |
| LLM API | Doubao/DeepSeek/Qwen | 主力模型 API 访问 | API Key 配置 | 按调用量计费 |

> **总计最低资源**：约 16C 30G + 100GB SSD（不含 GPU）

### 任务拆分

#### T1.1 企业安全网关搭建

**类型**：独立任务（无前置依赖）  
**预估工时**：3天

**范围**：
- 部署 Kong/APISIX 网关，配置 JWT 验签、限流、CORS
- 配置统一入口路由规则（/api/v1/chat, /api/v1/ocr 等）
- 实现租户上下文注入（从 JWT 解析 tenant_id、user_id、role 写入请求头）

**细节补充**：
- JWT 密钥管理：首期使用对称密钥（HS256），密钥存 Nacos，后续升级 RS256
- 限流策略：按租户 100 req/min，按 IP 200 req/min，防突发流量
- 健康检查端点：/health 不鉴权，供 K8s 探针使用
- 错误响应格式：统一 JSON 结构 `{ code, message, trace_id }`
- SSL/TLS：首期内网 HTTP，生产前必须配置 HTTPS

**验收标准**：通过 JWT Token 可访问受保护 API，无 Token 返回 401

---

#### T1.2 DeerFlow 核心引擎部署

**类型**：独立任务  
**预估工时**：5天

**范围**：
- 搭建 DeerFlow 2.0 + LangGraph 1.0 + LangChain 0.3 项目骨架
- 实现 `AgentFastState` 状态定义（精简版，仅含核心字段）
- 实现基础 LangGraph 工作流：`router → planner → executor → END`
- 接入 LLM 统一网关（支持 Doubao/DeepSeek 切换）

**细节补充**：
- 项目结构建议：`src/core/`（引擎）、`src/harness/`（护栏）、`src/skills/`（技能）、`src/tools/`（工具）
- LLM 网关首期简化：只做模型切换 + 超时熔断，不含完整能力注册表
- 状态持久化：首期使用 Redis Checkpointer，key 格式 `checkpoint:{thread_id}`
- 环境变量管理：模型 API Key、数据库连接串通过 Nacos 统一管理
- 日志规范：结构化 JSON 日志，必须包含 `thread_id`、`tenant_id`、`trace_id`
- 错误处理：所有节点异常必须映射到 `AgentFailureState` 三分类

**验收标准**：发送自然语言问题到 /api/v1/chat，能获得有效的 LLM 回复

---

#### T1.3 Semantic Router 基础版

**类型**：依赖 T1.2（接口依赖——需要 LangGraph 定义的 router 节点接口）  
**预估工时**：5天

**范围**：
- 部署 Milvus 向量库，创建意图向量 Collection
- 使用 `bge-small-zh-v1.5` 进行 query embedding
- 实现冷启动配置（阈值下调至 0.75，抽样率 20%）
- 实现 LLM fallback 兜底路由

**细节补充**：
- 向量维度：bge-small-zh-v1.5 输出 512 维，Milvus 配置 COSINE 度量
- 初始意图集：至少预置 `general_chat`、`invoice_ocr`、`knowledge_qa` 三个意图
- 每个意图至少准备 20 条种子样本作为向量库初始数据
- 冷启动退出条件：全局请求数 ≥ 1000 且最近 200 次影子验证不一致率 ≤ 15%
- 影子验证日志：写入 PG 表 `router_shadow_log`，供后续分析
- Milvus Collection Schema：`{id, intent, embedding, tenant_id, created_at}`
- **注意**：首期不实现二级路由和自动微调，仅做基础向量匹配 + LLM 兜底

**验收标准**：不同类型问题（闲聊/发票/知识查询）能正确路由到对应 Skill

---

#### T1.4 Token Budget 守护机制

**类型**：独立任务  
**预估工时**：3天

**范围**：
- 实现 `TokenBudget` 预算分配（simple/moderate/complex 三档）
- 实现 `TokenBudgetGuard`：Redis 幂等计费 + 70%降级告警 + 95%硬熔断
- 集成到 LangGraph 工作流中（每次 LLM 调用前后扣减 Token）

**细节补充**：
- Redis Lua 脚本：幂等计费脚本需要在部署时预加载（SCRIPT LOAD）以减少网络开销
- Redis 降级：当 Redis 不可用时降级到进程内计数，但必须打告警日志
- 指标上报：首期写入结构化日志，不强制对接 Prometheus/Grafana
- TTL 设计：预算 key 24h 过期，operation_id 集合同步过期
- 并发安全：多 Pod 部署时依赖 Redis 原子操作保证计数一致，本地计数仅临时容错
- 与 LLM 网关集成：在 LLM 调用 wrapper 中自动调用 `guard.consume()`

**验收标准**：Token 消耗可追踪、超 95% 预算时请求被拦截并返回明确错误

---

#### T1.5 Planner 防死锁机制

**类型**：依赖 T1.2（接口依赖——需要 LangGraph planner 节点）  
**预估工时**：3天

**范围**：
- 实现 DAG 环路检测（拓扑排序 DFS）
- 实现 Planner 自纠错：重试上限 2 次 + 携带错误上下文
- 实现 Critic 后路由：超时熔断（30min）+ 重试上限 3 次

**细节补充**：
- DAG 校验三件事：环路检测 + 依赖完整性校验 + 孤立节点检测
- 空 DAG 处理：Planner 输出空计划视为结构非法，计入 `plan_retry_count`
- 错误上下文格式：`last_planner_error` 必须是结构化字符串，告诉模型修正方向
- 幂等键设计：Planner/Executor 每个节点需生成唯一 `idempotency_key`
- 超时计时：从 `state.start_time` 开始，30 分钟全局硬熔断
- **Critic 首期简化**：首期 Critic 只做格式校验（输出是否符合 Schema），不做复杂业务校验

**验收标准**：生成含环路的 DAG 时被拦截；Planner 错误后能携带上下文自动修正

---

#### T1.6 PaddleOCR 发票解析服务

**类型**：独立任务  
**预估工时**：5天

**范围**：
- 部署 PaddleOCR PP-Structure 服务（Docker 容器化）
- 实现发票 OCR 统一入口 API（支持图片上传 → 结构化字段提取）
- 实现 Claim-Check 模式：图片 → OSS 存储 → 返回 `doc_uri`
- 将 OCR 封装为 LangGraph 可调用的 Tool

**细节补充**：
- 支持格式：JPG/PNG/PDF，单文件 ≤ 20MB
- 输出字段：发票号码、发票代码、开票日期、金额（不含税）、税额、价税合计、销方名称/纳税人识别号、购方名称/纳税人识别号
- 置信度阈值：单字段置信度 < 0.7 时标记为 `low_confidence`，提醒人工复核
- 首期不启用 PaddleOCR-VL 局部升维（留接口，二期实现）
- Claim-Check 实现：使用对象存储适配层（首期可用本地文件系统模拟 OSS）
- 异步处理：首期同步调用（单张发票 < 10s），不启用 Suspend & Resume
- Tool Schema 定义：`{ name: "invoice_ocr", params: { doc_uri: str }, returns: { fields: dict, confidence: dict } }`
- 错误处理：图片损坏/格式不支持返回 `HUMAN_REVIEW` 级别错误

**验收标准**：上传增值税发票图片，返回结构化字段 JSON，关键字段准确率 ≥ 90%

---

#### T1.7 知识库 Agent（第一个 Skill）

**类型**：依赖 T1.3（接口依赖——需要 Router 路由到该 Skill）  
**预估工时**：5天

**范围**：
- 实现基础 RAG 链路：文档上传 → 切片 → Embedding → Milvus 存储
- 实现知识检索 Skill：用户提问 → 向量召回 → Reranker → LLM 生成回答
- 租户隔离：向量检索强制注入 `tenant_id` 过滤

**细节补充**：
- 文档切片策略：按段落切片，单片 ≤ 512 tokens，重叠 50 tokens
- Embedding 模型：复用 bge-small-zh-v1.5
- Reranker：首期使用 bge-reranker-base，Top-K=5 召回后 Rerank 取 Top-3
- Milvus Collection：`skills_knowledge`，Schema 包含 `{doc_id, chunk_text, embedding, tenant_id, skill_id}`
- 文档格式支持：首期支持 txt/md/pdf，后续扩展 docx/xlsx
- 无结果处理：检索无结果时，返回 "未找到相关信息，建议联系人工客服"
- 引用溯源：回复中附上引用的文档片段 ID 和相关度分数

**验收标准**：上传知识库文档后，用户提问能获得基于文档内容的准确回答

---

#### T1.8 WAL 双写审计基础版

**类型**：独立任务  
**预估工时**：3天

**范围**：
- 实现 `LocalWALStore`：本地文件 WAL（fsync 保证落盘）
- 实现 `secure_audit_flow`：先写本地 WAL → 异步发送下游（首期写 PG 代替 Kafka）
- 实现决策证据链记录：工具调用、参数快照、结果快照

**细节补充**：
- WAL 目录：`/var/agent/wal/`，需保证目录权限和磁盘空间
- WAL 文件格式：`{entry_id}.json`，fsync 强制落盘
- 首期不引入 Kafka：审计数据先本地 WAL → 异步写 PG 表 `agent_audit_log`
- PG 表结构：`{ id, thread_id, tenant_id, entry_type, content_json, created_at }`
- Janitor 首期简化：不单独部署 DaemonSet，用定时任务扫描 WAL 目录重试
- 审计内容：记录决策摘要（非完整思维链）、工具调用参数、执行结果、规则命中情况
- 日志保留：PG 中保留 90 天，超期归档（首期手动归档）

**验收标准**：每次 Agent 执行都有审计记录，WAL 文件在成功同步后自动清理

#### T1.9 基础前端界面

**类型**：独立任务  
**预估工时**：5天

**范围**：
- 实现 Web 对话界面（SSE 实时推送）
- 实现发票上传入口（拖拽/选择文件）
- 实现 OCR 结果展示（结构化字段表格 + 原图预览）

**细节补充**：
- 技术栈：React/Vue + SSE（Server-Sent Events），不用 WebSocket（简化部署）
- 对话界面：支持 Markdown 渲染、代码高亮、表格展示
- 文件上传：前端直传 OSS + 回传 URI 给后端，避免后端内存压力
- 响应式设计：支持桌面端和平板端
- 错误展示：友好的错误提示（如 Token 耗尽、超时、需要人工审批等）
- Loading 状态：显示思考中动画，长任务显示进度提示

**验收标准**：用户可在浏览器中正常对话和上传发票

## 第二阶段：Harness 构建（M3-M4，8周）

### 阶段目标

构建完整的安全护栏体系和记忆系统，使 Agent 具备企业级安全控制能力。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 工具拦截 | 未授权工具调用 100% 被拦截，身份参数 100% 被强制覆写 |
| HITL 审批 | 高风险操作触发审批，审批超时自动否决 + 资源补偿 |
| 记忆连贯性 | 同一线程 20 轮对话后，Agent 仍能正确引用早期上下文 |
| Scratchpad 内存 | 超长推理链（100步）Redis 内存占用 < 50MB |
| OCR 融合 | PaddleOCR-VL 局部升维 POC 完成，模糊区域识别率提升 ≥ 10% |

### 新增中间件

| 中间件 | 用途 | 部署要求 | 最低资源 |
|--------|------|----------|----------|

#### T2.4 三层记忆模型

**类型**：依赖 T1.2（接口依赖——集成到 LangGraph 工作流）  
**预估工时**：8天

**范围**：

- L2 情节记忆：Event-Sourced Diffs + 快照检查点 + 事件流回放
- L3 语义记忆：Milvus 租户隔离检索 + 二次校验
- 融合召回：L1 滚动窗口 + L2 实体/摘要 + L3 语义召回权重融合

**细节补充**：

**细节补充**：

## 补充说明一：使用 DeerFlow 作为基座时，必须先明确的内容

下面这些不是“可选实现细节”，而是基于 DeerFlow 作为底座时，必须在实施方案里先写清楚的基线。否则任务接单后，开发、测试、运维、验收都会出现理解偏差。

### 1. 基座边界必须明确

- DeerFlow 在本项目中到底是“完整业务框架”还是“工作流编排内核”。
- 哪些能力属于 DeerFlow 基座，哪些能力属于本方案新增模块，哪些能力只做适配层。
- 是否允许直接修改 DeerFlow 核心代码，还是只能通过扩展点、插件、适配器实现。
- DeerFlow 的版本号、分支策略、依赖锁定策略是否固定。
- 现有仓库中与 DeerFlow 相关的目录、包名、命名规范、构建方式是否统一。

### 2. 执行模型必须明确

- 任务执行是同步、异步、还是混合模式。
- `router -> planner -> executor` 是否是唯一主链路，还是允许旁路工具流。
- 每个节点的输入、输出、失败态、重试态、超时态是否统一。
- `thread_id`、`run_id`、`task_id`、`trace_id` 的含义和生命周期。
- 是否允许长任务跨进程恢复，恢复点保存在哪里。

### 3. 状态模型必须明确

- `AgentFastState` 的字段集合、不可变字段、可选字段、版本号演进规则。
- 哪些状态保存在内存，哪些状态落 Redis，哪些状态落 PostgreSQL。
- Scratchpad、短期记忆、长期记忆、审计日志之间的边界。
- 状态快照如何回放、如何对比、如何做兼容升级。
- 状态丢失后是重算、回滚还是人工介入。

### 4. 工具与 Skill 体系必须明确

- DeerFlow 里 `Tool`、`Skill`、`Agent` 的职责边界。
- Skill 的注册方式、元数据格式、目录结构、版本发布规则。
- 工具调用是否要统一经过拦截器和审计层。
- 工具 schema 的定义语言是什么，JSON Schema、Pydantic、还是自定义协议。
- Tool 的幂等性、重试、超时、返回体结构是否统一。

### 5. 模型接入必须明确

- 支持哪些模型供应商，是否要求可热切换。
- 统一网关是否只是“模型代理”，还是还承担 prompt 模板、路由、限流、熔断。
- prompt 模板、系统提示词、工具提示词的存放位置和版本控制方式。
- 失败回退规则：模型失败后是切换模型、降级规则引擎、还是直接失败。
- 流式输出、截断输出、超长上下文的处理方式。

### 6. 记忆体系必须明确

- DeerFlow 是否默认有 memory 层，如果有，哪一层是框架能力，哪一层是业务能力。
- L1/L2/L3 的读写顺序、合并规则、覆盖规则、TTL。
- 记忆检索是“先检索后生成”，还是“先生成后回写”。
- 记忆污染如何防止，特别是跨租户、跨任务、跨回合。
- 是否允许从历史对话中自动提炼知识并写入知识库。

### 7. 安全与租户隔离必须明确

- 网关是否是唯一可信入口，内部服务是否一律不信任客户端 Header。
- 租户信息是由 JWT、mTLS、还是服务间签名下发。
- 细粒度授权是 RBAC、ABAC，还是二者并存。
- 敏感字段脱敏、Tokenization、反脱敏、审计可见性如何分级。
- Prompt Injection、越权工具调用、跨租户召回如何防护。

### 8. 可观测性必须明确

- 标准日志字段集合必须统一：`tenant_id`、`thread_id`、`trace_id`、`run_id`、`skill_id`、`tool_name`。
- 指标、日志、Trace、Eval 的归档规则是否统一。
- 哪些事件必须进入审计链，哪些事件只进业务日志。
- 是否需要回放能力，回放粒度到什么层级。
- 错误分类是否统一为 `RETRYABLE / HUMAN_REVIEW / FAILED_CLOSED / INTERNAL_ERROR` 之类。

### 9. 部署与运行时必须明确

- 是单体部署、模块化单体，还是按服务拆分。
- Docker 镜像是统一基座镜像，还是每个模块单独镜像。
- K8s 是否是唯一目标环境，是否支持本地开发和离线部署。
- Redis、PG、Milvus、OCR、LLM 网关是否允许本地模拟。
- 配置中心、密钥管理、证书轮换策略如何落地。

### 10. 测试与验收必须明确

- 单测、集成测试、回归测试、压测、红队测试分别由谁负责。
- 每个任务的验收是“接口可用”还是“业务样例通过”还是“性能指标达标”。
- 是否要保留黄金样例集、回放集、对抗集。
- 方案里的数值指标是否是硬门槛，还是阶段性目标。

## 补充说明二：基于 DeerFlow，建议在文档里列为“待讨论”的事项

这些内容可以先不在第一版定死，但必须明确为“待讨论”，否则接单时会默认成实现范围的一部分。

- LangGraph 与 DeerFlow 的关系：DeerFlow 是包裹 LangGraph，还是替代部分能力。
- `Skill` 是否必须是运行时可热更新，还是允许重启生效。
- 是否坚持 Redis Checkpointer，还是在后续阶段切到 PostgreSQL。
- `Milvus` 是否是唯一向量库，还是允许 PG Vector 作为轻量替代。
- `SSE` 是否足够，是否需要升级 WebSocket 或双通道事件总线。
- `HITL` 是强制人工审批，还是允许规则自动放行。
- OCR 是先规则后模型，还是先模型后规则。
- 审计是 PG 作为主库，还是 Kafka/ClickHouse 作为主链路。
- 是否要把“evaluation layer”做成独立平台能力。
- 多租户隔离是逻辑隔离优先，还是物理隔离优先。

## 补充说明三：任务接单前必须写清楚的通用模板

每个任务建议至少补充下面 6 项，不然接单人很难知道“到底要做到什么程度”。

1. `目标`：这个任务最终要解决什么问题。
2. `输入`：任务依赖哪些已有能力、接口、数据、配置。
3. `输出`：要交付的代码、接口、配置、文档、测试。
4. `边界`：明确本任务不做什么。
5. `验收`：用什么样的样例、指标、日志或测试来判断完成。
6. `依赖`：依赖哪些其他任务、哪些基础设施、哪些待定决策。

| T3.4 财务报销 Agent | 处理发票、科目映射、合规校验、凭证生成的完整链路 | 发票解析、科目映射、校验规则、批量处理、异常回退 | 不做总账记账系统，不做 ERP 全量替代 |
| T3.5 金融审批 Agent | 支持多维权限控制和审批流转，覆盖高风险金融操作 | ABAC 规则、审批链路、二人审批、审计记录 | 不做通用 BPM，不做所有部门的统一审批中心 |

### 第四阶段任务补充

| 任务 | 这件事到底要做什么 | 建议补充的交付物 | 必须写清楚的边界 |
|------|-------------------|------------------|------------------|
| T4.1 Skill 广场 + 上架审核 | 把 Skill 当成可发布、可审核、可回滚的产品资产管理 | Skill 规范、审核流、灰度发布、回滚方案 | 不做开放市场，不做外部生态分发 |
| T4.2 二级路由 | 在一级意图之下进一步做子域路由，提高命中精度 | 子意图定义、候选集过滤、向量检索、精确路由 | 不做无限层级路由，不做自动学习闭环 |
| T4.3 Token 成本租户视角监控 | 从租户维度看清 token 消耗和成本热点 | 租户报表、阈值告警、图表、导出 | 不做完整计费中心，只做成本可视化和告警 |
| T4.4 Evaluation Layer + 发布门禁 | 让发版前必须过评测，不达标不能发 | 评测集、回放器、门禁规则、CI/CD 集成 | 不做主观审核替代，也不做离线训练平台 |
| T4.5 全链路可观测 | 把推理、工具、记忆、校验都接入统一观测 | OTel 接入、Trace、指标、Dashboard、告警规则 | 不做单纯日志堆砌，要有统一追踪语义 |
| T4.6 生产规格扩展 | 把基础能力升级为更高可用的生产形态 | HA 架构、主从、哨兵、冷热点分离 | 不做一次性“大上云”，要保留平滑迁移路径 |
| T4.7 全链路压测 | 验证在目标并发和场景下系统是否能扛住 | 压测脚本、数据集、结果报告、瓶颈分析 | 不只看平均值，必须看 P95/P99 和错误率 |
| T4.8 蓝绿对抗 + 安全审计 | 通过攻击样本和对抗测试验证安全边界 | Prompt Injection 样本、越权测试、审计报告 | 不做一次性形式化安全检查，要有可复现样本 |
| T4.9 SR 稳定运行切换 | 当路由稳定后，把实验态切到常态配置 | 阈值调整、灰度规则、回退开关、观测指标 | 不等于“上线即永久关闭实验”，要能回滚 |
| T4.10 沙箱灰度启用 | 对高风险代码执行放入沙箱，降低执行风险 | 沙箱运行时、执行器、预热池、策略开关 | 仅覆盖高风险任务，不要一开始就全量沙箱化 |
