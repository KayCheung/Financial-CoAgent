# 第十四节：风险应对矩阵与生死线 (对应文档第14章)

**概览**：任何技术方案都有风险。这里不列举那些泛泛而谈的“怕断电、怕黑客”，它列举的全部是大模型特有在大型项目里的病灶，同时强行下了三道“生死令”。

## 2. 它的机制和原理是什么？（三道绝不妥协的生死线约束）

1.  **PII 脱敏绝对不准用普通的替换方案**
    *   绝不允许像咱们写简易替换代码那样“把文库中所有的某身份证替换成星号”。因为一旦有两串数字凑巧重复或混入财报里的数字，整个文本架构就错位乱套了。
    *   **强硬规定**：必须老老实实获取所有需打码数字在文本中的确切几行几列“经纬度”，然后抠掉对应的砖头，换个假砖塞进去。绝不让开发人员在这偷懒妥协。

2.  **记忆状态绝对不准无尽地滚键盘记录（需强制建立检查点）**
    *   当一个任务在服务器里等待审批了好几天，中间发生过成百上千条交互日志。如果在中途发生灾难要恢复状态时，绝不准系统像放长慢速VCR一样从头一帧一帧地读取上百页日志拼状态。
    *   **强硬规定**：必须阶段性“拍照存档（保存快照对象 Snapshotting state）”。恢复时，先拿出最近一张近照，再照着近照拼上后面少许的修改记录，火速恢复。

3.  **不设标准考试线的，绝不准上线**
    *   对于新做的一个处理财报能力，如果没有拿历史的财报题目和标准答案当质检员进行全线试跑，跑过了设定的死标准，即使大家开会都觉得它很好，也绝不准点发布按钮。

## 3. 还有其他什么技术方案？（业界对比）
业界针对风险管理的宽从对比：
1. **亡羊补牢型风险管理（Reactive SLA）**：很多系统的合规与恢复机制发生在发现问题后，大模型回复带有 PII 了然后再人工拉入黑名单。**缺点**：在强监管的金融与财税赛道，发生即是灾难。这种宽松管控不仅面临监管罚款警告，更会被客户直接停用。
2. **纯粹牺牲智能保安全（重度模板化）**：为了保安全，干脆把所有的意图强行锁死进死板的格式甚至不给大模型上下文权限。**缺点**：AI 退化为传统软件，丧失了涌现能力和灵活性。本方案试图探索“放开智能的大脑但死锁手脚与通道”的最佳边界点。

## 4. 中间件与组件选型分析

### 脱敏网关 / 规则检查引擎
*   **为什么用它**：在脱敏处理环节（生死线约束1），系统往往内嵌轻量前置文本处理器。基于明确坐标进行占位符插补。
*   **可选的替换组件**：可以使用基于 Presidio 等开源的安全过滤引擎，提供专为 PII 实体抽取和隐藏设计的底层能力组合；如果直接用大模型来进行前置脱敏则由于速度慢和易发幻觉（漏打或错打码）而不被本架构优先考虑。
