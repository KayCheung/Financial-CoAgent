# 第六节：Semantic Router (冷启动优化) (对应文档第6章)

**概览**：Router 的作用是“指路前台”。当用户提出一句“帮我查一下这季度财报”，系统得根据这句话判断到底该把任务交给哪个业务 Agent去执行。

## 1. 为什么不直接用大模型做 Router？
直接把用户的话抛给大模型让它选方向极度准确但问题是：**太慢（要思考好几秒）+太贵（每次都要烧钱）**。
每天海量的询问，必须有一个轻量极速的办法来分流。这就是语义路由器出场的原因。

## 2. 它的机制和原理是什么？（混合双轨制）

### 特效一：先便宜后贵，低头兜底算法
*   进来一句话，系统先找传统的“向量匹配库”（这玩意又快且0消耗），通过计算文本 Embedding 向量的余弦相似度，如果这个库信心满满（相似度 > 90%），直接交给对应的Agent干活，省去了大模型的时间。
*   如果信心极其不足（相似度 < 60%），证明题太难，才会降级去叫贵且聪明的大模型（作为高级判定仲裁器）来当大脑判定类别。

### 特效二：影子验证 (监控错题本)
*   最精彩的一环是对于处于“半懂不懂”的中间分数地带，或是系统刚启用不久（冷启动期）。系统表面上按“低级预测”先把路指了，但背后偷偷再找一次大模型（大算力），比对当初自己指路对不对。
*   如果比对出错了，“错题”不会直接改系统，而是被自动打好标签塞给管理员。**等到错题本攒多了，进行微调强化**。这就是所谓的带有“Domain Shift”感知能力的自动进化。

## 3. 还有其他什么技术方案？（业界对比）
业界的传统意图路由手段有：
1. **纯 LLM Function Calling（函数调用路由）**：把所有的路线选择写进 Prompt 叫大模型自己选。**缺点**：太慢（增加上千毫秒延迟）且费用高企，如果路由分支多到几十个，极易造成上下文混乱超载。
2. **基于硬编码正则与关键字匹配**：非常快，但极其僵化。一旦用户表达法稍微变体（如“查查本季度收入”换成“帮我看下Q3进账”），关键字就接不住了，维护成本高。混合双轨制结合了两者的优点。

## 4. 中间件与组件选型分析

### Embedding 模型与轻量向量查询引擎 (Faiss 或同类库)
*   **为什么用它**：在 Router 这个环节，查询并发是最顶格的（所有流量入站第一关）。不需要沉重的集群级存储方案，只需要在内存里做一个极致神速的 KNN 近似查询（Nearest Neighbor）。比如 Faiss 单台机器就能吃下极高的 QPS，确保不在此处拖慢用户响应。
*   **可选的替换组件**：可以直接复用系统全局的 Milvus 或 PostgreSQL+pgvector。如果追求极简甚至本地单机纯内存字典或基于轻巧的 ChromaDB 也可以担纲此任。大模型层面则可选择较低维度的 Embedding 接口以确保速度（如 BGE 系列）。

## 总结
这套网关兼顾了小作坊的极度便宜与大模型的极度聪明，它不是为了次次保证对，而是把错的概率圈进了可以学习与报警的防线内。
