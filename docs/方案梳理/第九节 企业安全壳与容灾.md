# 第九节：企业安全壳与容灾 (对应文档第9章)

**概览**：一切关于合规、不丢失记录、不出核心数据隐患的技术底线，都包在这一层“壳”里。也是银行等金融机构最看重的基础条件。

## 1. 最怕什么灾难？
这不比寻常应用。最怕停机断电时正在进行中的高危审批记录不翼而飞（无对账证据）；最怕员工手机里的客服记录一旦发给云端AI，带上了用户的真实身份证号（PII泄露风险）。

## 2. 它的机制和原理是什么？（三大保命机制）

### 2.1 WAL 黑匣子级审计录像（绝对不丢日记）
*   普通日志如果在往数据库传输途中突然网断了可能就丢了。本方案采取老旧且最稳妥的先决操作：写进本地硬盘（带fsync强制落盘，不写完不回执），然后再向远端库异步上传。
*   即使系统被雷劈死机了，只要硬盘没烧，还有一个独立的 Janitor 进程扫地机器人，重启后会把硬盘里没上传成功的日志原封不动地补传到远端。这叫“绝无可能出现无记录的黑户操作”。

### 2.2 双向脱敏网关 (基于坐标的高级打码)
*   **进站打标**：用户输入了一大串话里含个身份证号。系统发现后，不是普通的找字替换（找字替换极其容易发生文本扰乱），而是根据精准“坐标位置”，把它挖下来存起来，然后用一串乱码（如 `[ID_CARD_7A9B]`）代替塞回原话去。这样扔给云端的大模型，它只能看到这串乱码。
*   **出站按身份还原**：大模型推断完毕传回结果时，网关看看坐在屏幕前的你是基层员工还是高管。如果是基层，只给你看“前四后四”加星号的掩码；如果是特权人员，才把刚才存在本地的数据严丝合缝按坐标回填显示给你。

### 2.3 降级回落机制
*   这套系统由于大量用了 Redis 作为中间神经传导（如快照状态）。假如 Redis 宕机了呢？不用怕，底层还有 PostgreSQL 数据库的实体记录作为终极降级靠山，保证宁愿慢一点，不会当场死亡停摆。

## 3. 还有其他什么技术方案？（业界对比）
业界关于合规与容灾的处理，有如下对比方案：
1. **纯同步数据库（Synchronous DB Commits）**：要求每一个请求都在主库双写完毕并且从库也接收完毕才返回。**缺点**：在 AI 这种高吞吐量且经常需要传输数 MB 对话历史的环境下会把主库网络带宽打爆，且延迟巨大。所以引入 WAL 本地先决才是王道。
2. **简单的正则字符替换（Naive Regex Replace）脱敏**：传统的脱敏都是直接正则匹配并替换。**缺点**：很容易将文章中和身份证号形似的订单编号勿伤搞混导致大模型解析崩溃，基于绝对“坐标截词”的二次还原方法虽然重但极稳。

## 4. 中间件与组件选型分析

### 本地存储 (磁盘级 Write-Ahead Log) 与 Janitor 进程
*   **为什么用它**：面对极端系统灾难（网线被拔、系统宕机），只要硬件磁盘还在，带 `fsync` 强制刷新到底层的本地日志文件就是唯一能“捞人”的黑匣子装置。
*   **可选的替换组件**：Kafka 的 Producer 带 `acks=all` 本质上也是依靠多磁盘冗余，如果预算高可以直接依赖 Kafka 集群；但在最穷凶极恶的环境隔离和单节点崩溃中，本地文件 WAL 最为坚固。

### 核心数据库与缓存 (PostgreSQL 与 Redis)
*   **为什么用它**：Redis 作为高速通道，负责状态转场和临时票据极速查询；PostgreSQL 通过丰富的数据类型与坚如磐石的稳定性作为兜底。
*   **可选的替换组件**：Redis 的高可用架构可以用 Memcached 配合其他持久化补充；PostgreSQL 也可替换为 MySQL 或 OceanBase 这类核心国产金融分布式库，后者在多中心多活（Multi-Active）容灾上更自带光环。

## 总结
这是全套设施中的金丝软甲。它的使命非常朴素并且不带有任何 AI 相关光环——那就是留记录、打掩码、不宕机。却是在金融环境落地的敲门砖。
