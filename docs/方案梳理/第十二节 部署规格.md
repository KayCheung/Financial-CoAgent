# 第十二节：部署规格与技术栈 (对应文档第12章)

**概览**：整套系统并不是非要买一百台服务器才能转得起来，架构本身像变压器一样具备非常高的伸缩弹性。

## 2. 它的机制和原理是什么？（弹性的三级伸缩机制）

### 部署的分为三等模式：

1.  **极简小作坊版（规格A）：适用于老板想看POC效果，5个人小团队。**
    *   能砍的周边防爆设备全砍了。不需要庞大的ClickHouse集群来支持万年不挂的审计功能，普通关系数据库 PG 顶替；那些重型杀伤性的极度防爆沙箱虚拟机不运行了，大家都彼此相信是“内信任环境”。

2.  **企业稳步上线版（规格B）：适用于内部推广，需要 8-9 人。**
    *   该有的都有，把核心拦截 Harness 的功能上全，普通级别的隔绝容器跑起来。它能应付日常没有故意破坏环境下的高压任务。

3.  **金融纯血多活版（规格C）：涉及核心票款审批、高并发，需十多名资深架构师撑场。**
    *   万物皆为强隔离，核弹级安全沙箱全速开启，各种Redis集群、Patroni热备机制上线防止连网线被挖断。它必须像银行后台一样，任何时刻不能出死锁不能漏一条审计残渣。

## 3. 还有其他什么技术方案？（业界对比）
业界的架构部署策略流派：
1. **重度绑定公有云全家桶方案**：直接利用公有云厂商（如阿里云、AWS）的托管 AI 服务和云服务全家桶。**缺点**：金融机构命脉数据直接面临公有云审查风险和 vendor lock-in（供应商锁定）。本方案主张“私有化底座自由搭配”，可随时拔网线。
2. **僵化的巨石单体应用架构（Monolithic）**：不分级别的将大模型所有逻辑固化在一个系统中。**缺点**：启动重，一损俱损（比如向量库挂了连天气都查不了）。三级弹性部署能做到“没钱就跑单机，有钱就上战列舰”。

## 4. 中间件与组件选型分析

### API 流量大将 (Kong 网关)
*   **为什么用它**：在规格C的纯血统并发下，几千万次包含租户鉴权的请求必须在一瞬间分发完毕，Kong 基于 Nginx 极其硬核的处理能力和插件扩展性使得其在金融网关层几乎是王者。
*   **可选的替换组件**：APISIX 是性能更高且支持多语言编排的国产新星；或用更轻量的 Envoy（更契合 K8S 服务网格生态）。

### 异步抗压血管 (Kafka 消息队列)
*   **为什么用它**：它在规格C下能接住诸如“每秒上万条审计日志”、“巨幅账单核算延迟队列”等暴突发流量，防止后面的慢速消费者被打穿。
*   **可选的替换组件**：RabbitMQ 较轻，延迟极低但不适合做历史重放和极限高吞吐；Pulsar 则作为后起之秀拥有存算分离优势。

### 数据库高可用 (Patroni PG 集群)
*   **为什么用它**：PostgreSQL 本身是单机的，Patroni （搭配 etcd 或 ZooKeeper）组成了自动故障转移和脑裂防护的集群，这就是“连网线被挖断”都不怕底气。
*   **可选的替换组件**：如果全部倒腾到商业版的分布式库如 TiDB 或 OceanBase，可以直接获得内置的高可用能力。

## 总结
这套架构在技术选型上非常明确：抛弃花坛上五彩斑斓但不扛造的各类“新兴潮玩AI小工具库”。极其硬汉式地采用了类似 Kong 网关、Kafka 队列、Patroni 集群这种传统但经过亿万次金融生死考验的工业级传家宝来作为地基承接上方大模型的骚套路。
