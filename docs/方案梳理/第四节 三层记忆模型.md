# 第四节：三层记忆模型 (对应文档第4章)

**概览**：为了防止大模型在长线任务中“前言不答后语”，并且保护同一平台上企业租户各自的数据隐私，我们给 Agent 植入了“三层记忆体系”。

## 1. 为什么需要特制的记忆模型？
普通的大模型对话，直接把历史聊天记录硬塞回去就好了。但在企业场景里：
1. **记不住**：大模型记性有限制（窗口有限），资料太多会被撑爆。
2. **会篡改**：如果有几千字的复杂财报，大模型有时候瞎总结，导致重要的金额和发票号静默丢失或改写。
3. **数据窜台**：如果不做底层物理阻断，A租户在提问时可能会意外查出B租户的合同。

## 2. 三层机制 (怎么实现的？)

### L1 即时记忆轨 (短期工作台 - Redis 滚存)
*   就是大模型“现在的随笔本”。采取分片存入内存条（Redis），最多保存最近20步的原文，太早的或者太大的内容只留一个“指纹”提取码，防止把它自己撑死。会话一关，随即销毁。

### L2 情节记忆 (中期保险柜 - 增量日志 Event-Sourced)
*   **绝对禁止模型瞎总结！** 这是这一层最重要的原则。过去喜欢让模型精简状态，很容易把关键的金融数字精简没。因此这里采用事实留痕：**“只记录状态增加和改变的部分（Diff）”**，用便宜的冷存设备换取绝对的安全。出了错随时可以像放录像带一样回放当时的进展，数据100%保真。

### L3 语义记忆 (长期经验库 - Milvus 向量库)
*   大模型去学过的技能、企业过往的长知识都在这里。
*   **五步脱敏与硬隔离防线**：这是重中之重。系统不信任大模型的请求，会在API网关层、拦截层、数据库层、向量库等 5 层面**强行塞入当前员工的租户标识**。绝不允许大模型通过所谓的“提问技巧”骗过系统去翻阅别家公司的数据。

## 总结
这套三层记忆系统，既保证了大模型“当前聊得顺畅（L1）”，又给一切操作留了“监控录像且不许大模型自己删改（L2）”，还把长篇大论的资料“安全地锁入私人储物柜绝不窜台（L3）”。
