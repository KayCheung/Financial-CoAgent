# 第四节：三层记忆模型 (对应文档第4章)

**概览**：为了防止大模型在长线任务中“前言不答后语”，并且保护同一平台上企业租户各自的数据隐私，我们给 Agent 植入了“三层记忆体系”。

## 1. 为什么需要特制的记忆模型？
普通的大模型对话，直接把历史聊天记录硬塞回去就好了。但在企业场景里：
1. **记不住**：大模型记性有限制（窗口有限），资料太多会被撑爆。
2. **会篡改**：如果有几千字的复杂财报，大模型有时候瞎总结，导致重要的金额和发票号静默丢失或改写。
3. **数据窜台**：如果不做底层物理阻断，A租户在提问时可能会意外查出B租户的合同。

## 2. 它的机制和原理是什么？

### L1 即时记忆轨 (短期工作台 - Redis 滚存)
*   就是大模型“现在的随笔本”。采取分片存入内存条（Redis），最多保存最近20步的原文，太早的或者太大的内容只留一个“指纹”提取码，防止把它自己撑死。会话一关，随即销毁。

### L2 情节记忆 (中期保险柜 - 增量日志 Event-Sourced)
*   **绝对禁止模型瞎总结！** 这是这一层最重要的原则。过去喜欢让模型精简状态，很容易把关键的金融数字精简没。因此这里采用事实留痕：**“只记录状态增加和改变的部分（Diff）”**，用便宜的冷存设备换取绝对的安全。出了错随时可以像放录像带一样回放当时的进展，数据100%保真。

### L3 语义记忆 (长期经验库 - Milvus 向量库)
*   大模型去学过的技能、企业过往的长知识都在这里。
*   **五步脱敏与硬隔离防线**：这是重中之重。系统不信任大模型的请求，会在API网关层、拦截层、数据库层、向量库等 5 层面**强行塞入当前员工的租户标识**。绝不允许大模型通过所谓的“提问技巧”骗过系统去翻阅别家公司的数据。

## 3. 还有其他什么技术方案？（业界对比）
业界通常存在两种典型做法，与本方案的“三层模型”形成对比：
1. **单一长上下文硬塞（例如原生 GPT-4 128k 窗口处理）**：将所有历史记录甚至文档全部放入一次对话提示词中。**缺点**：极其昂贵，速度随着对话变长呈指数级下降，且每次调用都在消耗不必要的资源。
2. **粗暴全局向量检索（通用 RAG 方案）**：将所有企业文档和对话存入一个毫无边界的向量库中，每次提问强行检索Top K。**缺点**：缺乏多租户物理隔离和精准匹配，极易将A公司的发票数据回答给B公司，且往往会丢失细粒度的数字级精确度。

## 4. 中间件与组件选型分析

### Redis (用于 L1 即时记忆轨)
*   **为什么用它**：Redis 作为高性能的内存数据库，对读写极速响应非常关键。大模型的思考速度本就较慢，周边的记忆抓取必须达到亚毫秒级，Redis 支持高速滚存与过期淘汰（TTL），完美契合“随笔本”即用即抛的特性。
*   **可选的替换组件**：Memcached 也能做到类似纯内存缓存；或者如果在小微规模下，可以直接使用应用内的进程内存（如 Caffeine 缓存），但在多租户分布式集群环境下，Redis 仍是行业事实标准。

### Event-Sourced 增量日志 (用于 L2 情节记忆)
*   **为什么用它**：核心是为了“不可篡改”和“完美溯源”。放弃普通的 UPDATE 覆盖模式，采用只追加（Append Only）的日志流系统，在发生金融核对争议时能够精准复原当时的任务状态。
*   **可选的替换组件**：Kafka/Pulsar 可以用来做事件流存储；或者通过 PostgreSQL 配合严密设计的 Append 历史表来落地。

### Milvus (用于 L3 语义记忆)
*   **为什么用它**：Milvus 专为大规模向量数据设计，提供优秀的混合查询过滤（如 `tenant_id=A` 同时 `vector_distance<0.5`），这是实现多层权限硬防脱敏隔离的刚需；且具备强可扩展性。
*   **可选的替换组件**：Qdrant、Pinecone、甚至是加装了 pgvector 插件的 PostgreSQL。如果是中小规模或低成本考量，可以直接用 PostgreSQL+pgvector 实现降本增效；若需顶级性能托管，可选 Pinecone。

## 总结
这套三层记忆系统，既保证了大模型“当前聊得顺畅（L1）”，又给一切操作留了“监控录像且不许大模型自己删改（L2）”，还把长篇大论的资料“安全地锁入私人储物柜绝不窜台（L3）”。
