# 第八节：Skill 中心 (生命周期治理) (对应文档第8章)

**概览**：为了让大模型在特定专业场景（例如：专门审合同、专门报销开票）有稳定的发挥，平台引入了“Skill（技能点）”的概念，也就是给大模型打包好的特定武器库和操作手册，供它接单调用。

## 1. 为什么需要治理 Skill？
如果不加管理地向系统里投放各种Prompt提示词和API，今天大模型还能看懂发票，明天系统稍微升个级或者更新一下提示词哪怕加个逗号，它可能就满嘴跑火车认不出字了，我们称之为“模型漂移”。这就需要一套极其严峻的类似“苹果应用商店App上架审核机制”。

## 2. 它的机制和原理是什么？（红队机制与专项路由）

### 2.1 上架审核与红队扫描（防后门）
*   任何工程师给平台写好一个“自动查账Skill”准备发布前。必须被系统进行“红队压测”（比如用10种黑客最爱的攻击提示词套它，看它会不会违规暴露余额）。还要根据历史评估它的花钱水平。之后才被允许小规模灰度给真实用户。

### 2.2 二级路由防过载
*   公司一旦建了上千个Skill，大模型很容易眼花迷路。于是加入了元数据过滤：你要用某个 Skill？先检查这把钥匙当前部门有没有资格配，没资格连看都不让你看到，缩小搜索圈。

### 2.3 专项 OCR 混合架构（重器压阵，防止纯视觉模型翻车）
面对金融场景里极其复杂难懂的带有乱涂乱画的财报表格，绝对不能单纯指望像 GPT-4o 这种纯视觉模型的眼睛（它非常容易看错行导致金额串行）。
*   **传统版面为主**：必须坚持先用传统老派且绝对严谨的经典 OCR 引擎（PaddleOCR）去一笔一划识别结构。
*   **大模型降维辅佐**：只有遇到模糊的盖有印章遮挡的数字等老办法认不清的碎片区域时，才切出这一块“碎片”紧急召唤多模态大模型的眼睛（PaddleOCR-VL）去依靠上下文推理出是什么字。

## 3. 还有其他什么技术方案？（业界对比）
业界针对技能（Skill/Tool）管理的常用方案：
1. **类似 OpenAI GPTs Store 模式**：自由上架，无需硬性红队拦截验证，所有 Prompt 直接交给大模型自己看心情调用。**缺点**：太随意，一旦某个提示词引起冲突或陷入死循环，没有任何防御机制，极易发生“模型漂移”甚至信息泄露。
2. **纯粹多模态大模型“看图说话”（取代OCR）**：如直接将图片发给 GPT-4V 盲解。**缺点**：在金融数字这种密度高且“失之毫厘谬以千里”的场景，纯视觉大模型产生幻觉导致数字漏读的概率非常高，且成本翻倍。

## 4. 中间件与组件选型分析

### OCR 引擎 (PaddleOCR 及 PaddleOCR-VL)
*   **为什么用它**：PaddleOCR 是目前开源界极其强大的成熟系统，尤为擅长中文长文本、复杂表格还原以及盖章干扰。它的基于传统版面的识别逻辑极其稳定。而遇到需要意译的部位，选用专门为光学字符设计的 VL多模态模型（而不用通用大模型）能做到算力的精准投放。
*   **可选的替换组件**：Tesseract OCR（开源老牌但对中文复杂表格的支持相对疲软），或是对接阿里云/腾讯云商业版 OCR 接口（准确度极高但存在数据必须出公网的安全合规障碍，这也是自建专网 Agent 选择开源 PaddleOCR 的核心原因）。

## 总结
这套中心是对大模型各项专项专业本领的“纪律管控局”。不准私自在毫无监控的情况瞎改技能设定，且把对于金融文本的理解大权重放回了传统的确定性框架手里。
