# 第七节：沙箱执行层 (分级隔离) (对应文档第7章)

**概览**：在确认任务无害之后，大模型真正呼叫 API 或运行脚本大显身手的阶段，这层就是给大模型修筑的工作间（防爆网）。

## 1. 为什么做沙箱？为什么不所有调用都上强沙箱？
给大模型写 Python 脚本去解析文件或核数的权力，等同于在服务器上开后门。一不小心脚本无限死锁或者带木马，系统直接爆炸。
但如果什么小动作（比如仅仅是读个天气，拉取一下内网普通报表 API）都去启动沉重昂贵的“防爆沙箱虚拟机（比如gVisor等K8S环境）”，会导致整个系统极度的卡顿和迟缓。

## 2. 它的机制和原理是什么？（三级防爆体系）

系统在工具分工上不再一刀切，而是设定了三种隔离环境：

*   **L1 (API 直通 - 无隔离)**：如果这步只是调用公司本来就已经做得很完美、自带鉴权与限流的内网 API系统，由网关放行，极其轻量级，毫秒级通过。
*   **L2 (受控普通容器 - 软隔离)**：比如普通文件转换操作。仅仅依赖 Docker 中的 cgroup 限制与 Capabilities 剪裁，把进程关在阉割了网络和挂载权限的命名空间（Namespace）盒子里执行，便宜够用。
*   **L3 (强沙箱 K8S/gVisor - 核爆级隔离)**：面对最高危操作——即大模型现场编写的 Python 探测脚本，或加载未知的三方库——系统调动具有微型独立内核拦截虚拟化的沙箱。即使黑客攻破了沙箱内的环境，拦截层也会阻断向宿主机发出的大部分系统底层调用。

## 3. 还有其他什么技术方案？（业界对比）
业界面对 LLM 动态代码执行的方案对比：
1. **纯粹裸跑 / 普通 Docker 容器**：很多厂商直接开辟一个 Python 容器来执行大模型生成的代码。**缺点**：容器共享宿主机内核，一旦产生逃逸漏洞（由于常常需要Root权限处理依赖），整个服务器全部被黑或者资源会被死锁脚本占满枯竭，对于防守要求极高的金融机构不可接受。
2. **Serverless 无服务器函数（如 AWS Lambda）**：云端统一托管每次分配实例，非常安全。**缺点**：但对于企业极度敏感的财报数据来说，不仅涉及到云端脱壳隐私外泄的问题，也不能支持过于复杂的包含大体积依赖的长期运算，同时还会面对“冷启动”的调用超时折磨。

## 4. 中间件与组件选型分析

### Kubernetes (K8S) 与 gVisor 或 Firecracker
*   **为什么用它**：在 L3 防火墙级别，常规的 Docker 不够看。gVisor 通过一套在用户态重写系统调用（System Call）的边界保护层，或者 Firecracker 直接启动轻量级微虚拟机（MicroVM），实现了真正的进程隔离和彻底阻断，能安心跑大模型现场写出的“黑盒代码”。K8S 则是这整套沙堡集群最佳的伸缩和调度大总管。
*   **可选的替换组件**：如果不使用 gVisor/Firecracker，还可以选择开源界较新的 WebAssembly (Wasm) 运行时（如 Wasmtime）。Wasm 沙箱启动速度是纳秒级，极其轻巧安全，作为新兴方向潜力巨大，但目前缺点是对大量 Python 原生库（含 C 扩展的数理化包库）生态的支持依然较弱，正在迅速追赶。

## 总结
这层秉承着“按需动刀”的思想。能用旧API系统挡枪的，绝不滥用沙箱；唯独针对那种未知不可信的代码执行领域，给予最高规格的硬件级防御。兼顾了速度、成本和绝不妥协的底线安全。
