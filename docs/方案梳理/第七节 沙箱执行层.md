# 第七节：沙箱执行层 (分级隔离) (对应文档第7章)

**概览**：在确认任务无害之后，大模型真正呼叫 API 或运行脚本大显身手的阶段，这层就是给大模型修筑的工作间（防爆网）。

## 1. 为什么做沙箱？为什么不所有调用都上强沙箱？
给大模型写 Python 脚本去解析文件或核数的权力，等同于在服务器上开后门。一不小心脚本无限死锁或者带木马，系统直接爆炸。
但如果什么小动作（比如仅仅是读个天气，拉取一下内网普通报表 API）都去启动沉重昂贵的“防爆沙箱虚拟机（比如gVisor等K8S环境）”，会导致整个系统极度的卡顿和迟缓。

## 2. 怎么权衡解决（三级防爆体系）？

系统在工具分工上不再一刀切，而是设定了三种隔离环境：

*   **L1 (API 直通 - 无隔离)**：如果这步只是调用公司本来就已经做得很完美、有限流的内网 API。不用套沙箱了，极其轻量级，毫秒级通过。
*   **L2 (受控普通容器 - 软隔离)**：比如普通文件转换操作。扔进普通的阉割了权限的小盒子里去执行，便宜够用就行。
*   **L3 (强沙箱 K8S/gVisor - 核爆级隔离)**：只有当遇到高危工具——由于需要执行一段由大模型现场写出的不被信任的代码，或者需要跑用户传上来的不明脚本时——系统才会调动这种成本极高、启动又慢、但密不透风的纯血沙箱。

## 总结
这层秉承着“按需动刀”的思想。能用旧API系统挡枪的，绝不滥用沙箱；唯独针对那种未知不可信的代码执行领域，给予最高规格的硬件级防御。兼顾了速度、成本和绝不妥协的底线安全。
