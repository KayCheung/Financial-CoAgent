# 第五节：HITL 三段式审批状态机 (对应文档第5章)

**概览**：大模型做执行的时候难免需要申请特权（比如：“我算完账了，请问可以直接付款吗？”），这就需要真人来点击“同意/拒绝”。这就是 HITL（Human-in-the-Loop，人机协同）的核心。

## 1. 为什么需要这套复杂的审批机制？
如果大模型发出审批请求后，就一直在服务器里死板地挂机等待审批人点同意，一旦老板去开会了（等了5小时），这5个小时系统资源就被白白白占用了如果并发多了直接把节点憋死。而且如果彻底遗忘了，审批卡片就成了“僵尸”。

## 2. 它的机制和原理是什么？

### 三段式状态运转 (提交单据 -> 挂起收配给 -> 等待回传)
*   **不硬等**：当大模型发出“需审批”通知时，状态机先给员工（比如财务主管的飞书/企微）发送互动卡片。随后，系统释放掉大模型当前正在执行的线程和服务器计算资源，将其进入冬眠（序列化后状态写入关系型数据库）。

### “超时有感知”与资源退回 (最亮眼的设计)
*   **超时否决**：如果风险级别高的审批1个小时内主管没看，普通级别1天没看，不用干等，系统会通过定时心跳**“强行自动定调为否决”**。
*   **资源补偿（退货）**：更重要的是，大模型在申请审批前，可能已经在云端拉起了一个强大的沙箱（虚拟机环境）备用。如果审批被超时否决，状态机会立刻执行“扫地工作”，把所有预占的沙箱毁掉退还配额，并把各种占空间的临时文件清空，绝不留一点资源垃圾泄漏。

## 3. 还有其他什么技术方案？（业界对比）
业界通常处理人机协同（HITL）的方法有：
1. **同步阻塞等待（Synchronous Polling）**：最粗暴也最常见。大模型线程开启一个 while 循环不断轮询数据库看老板审批没有。**缺点**：极度浪费资源，一旦老板请假，线程就一直卡死在服务器里，几百个并发就能把后端打崩。
2. **简单的消息队列回调（单向触发）**：只管发送，利用 MQ 接收。但是它缺乏“全生命周期”的精细清扫能力，审批不通过或过期了往往就把记录扔在那儿变成孤儿数据。而本方案特有“超时资源补偿”能保证系统的纯净。

## 4. 中间件与组件选型分析

### 关系型数据库 (状态快照存储，如 PostgreSQL)
*   **为什么用它**：在“冬眠”环节，我们需要保证任务状态百分百准确地持久化，不可丢失。关系数据库对 ACID（原子性、一致性、隔离性、持久性）的支持天然适合这种状态机的严谨运转。
*   **可选的替换组件**：可以使用 Redis 作短暂持久化（开启 AOF），但数据价值高且要求持久的情况下，MySQL 亦可替代 PostgreSQL，MongoDB 也行但事务控制相对更麻烦。

### 分布式调度系统 (用于超时控制与心跳)
*   **为什么用它**：为了实现“高风险审批1小时强制否决”，需要用到精确的延迟队列或轮询调度器，来唤醒和扫描那些“冬眠”的状态。
*   **可选的替换组件**：可以使用 Redis 的 Delay Queue，或专用的分布式调度平台如 Quartz、Zookeeper 配合任务引擎，甚至 RabbitMQ/Kafka 的死信队列功能也能实现延迟触发。

## 总结
它不单单是一个在飞书发钉钉发卡片的小工具，它是给大模型的每次“借权操作”设定了严厉的**超时退款退票机制**，死死护住系统的昂贵计算量不被白占。
