# 第三节：规划层：防死锁编排引擎 (对应文档第3章)

**概览**：在第二节 Harness 层拦截了越权和破产风险后，系统终于允许大模型开始干活了。但大模型的工作方式像脱缰野马，如果不加控制，它很容易在一个牛角尖里无限死循环。因此，引入基于 DAG（有向无环图）和状态机的“规划层”就是为了给大模型铺设铁轨，防死锁、防迷路。

## 1. 为什么需要规划层？
普通大模型 Agent 解决复杂任务时，往往是“走到哪算哪（即无脑循环思维）”。经常会发生任务 A 依赖任务 B，任务 B 又想去等任务 A 的荒谬死锁情况。
在企业级场景，我们需要一个类似于“铁路调度中心”的明确状态机（使用 LangGraph 实现），强行规定每一步流转的合法性。出错了可以回滚，不会一直挂在那里白白燃烧 Token 经费。

## 2. 它的机制和原理是什么？(三大核心机制)

### 2.1 极简状态管理 (Fast State 与大对象行李寄存)
*   **机制原理**：系统规定，大模型运转的中心“工作台（State状态机）”上，**绝对不允许放体积庞大的物件**（比如大表单、几十MB的PDF原件、超长日志）。
*   **作用（Claim-Check寄存模式）**：所有重型数据强制存入外部仓库（如对象存储 OSS），工作台上只留一张写着提取码的领物小票（URI 链接）。这样一来，状态机的每一次扭转判断都轻如飞燕，不会发生内存炸挂。

### 2.2 防死锁 DAG 检测（给计划扫描错漏）
*   **机制原理**：大模型扮演的“项目经理（Planner）”在下达任务清单时，系统并不是直接听命去执行的。系统在此刻会切回传统的数学算法（DFS深度优先图遍历），把大模型的计划扫描一遍。
*   **作用**：如果扫描发现大模型制定的计划里出现了“衔尾蛇的圈圈（死循环环路）”，系统直接否决该计划并退回。如果大模型重试 2 次依然写出了死锁的计划表，此时系统会判断大模型智商掉线，立刻拉闸熔断终止任务。

### 2.3 失败状态强制收敛 (AgentFailureState)
*   **机制原理**：平时写爬虫或跑大模型，它的报错千奇百怪无章可循。但在金融 Agent 平台，它被硬性规定：无论什么天灾人祸，最终抛出的错误必须被归纳到下面 3 个框里（不允许出现未知的挂起状态）：
    1.  **可重试 (RETRYABLE)**：网络小抖动，退回原点重试 (最多3次)。
    2.  **强制人工介入 (HUMAN_REVIEW)**：遇到识别不清的模糊账单，大模型不敢背锅，机器直接定格暂停，并给员工企微发卡片求救确认。
    3.  **硬性熔断终止 (FAILED_CLOSED)**：核心数据库真挂了或者预算烧光了，大罗金仙来了也没救，直接关闭线程结束掉，防止僵尸进程。

## 3. 还有其他什么相关的技术方案？
*   **备选 A：纯模型驱动计划 (如知名开源项目 AutoGPT 早期版本)**
    *   原理：连循环也不管，遇到死锁就让大模型自己看 log 猜怎么解。
    *   缺点：大模型根本猜不对，只会无底洞一样刷 Token 报错。企业级绝对禁止这种“无兜底”模式。
*   **备选 B：全静态连线工作流 (如传统 RPA)**
    *   原理：完全由程序员拖拉拽把 100 步全部连好死框框。
    *   缺点：失去这层灵活性，无法适应一张财报发票缺少公章时的额外推理跳转，属于太死板的左倾方案。

## 总结
这套规划层，是用经典的计算机科学图论与状态机（硬规则），套住了大模型的想象力（软智能）。实现了既让它自己做计划排期，又能在它犯糊涂“画死结”时一巴掌扇停它，确保了金融架构最需要的**确定性流转**。
