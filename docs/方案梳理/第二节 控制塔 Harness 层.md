# 第二节：控制塔 Harness 层 (对应文档第2章)

**概览**：Harness 层是架构中的“硬骨头”与“安全守门员”。所有关于安全、成本和权限控制的业务逻辑都在这层兜底，它是凌驾于大模型之上的“一票否决”关卡。

## 1. 为什么需要 Harness 层？
大模型本质上是一个不可靠的文本生成器。如果直接让大模型调用企业核心接口（如OA系统、财务审批系统），会带来极高风险。Harness 层的作用就是“防呆不防贼”，拦截大模型的危险、越权、死灰复燃式的消耗行为，保证系统的“安全与成本底线”。完全指望大模型自我纠错是不切实际的，必须有传统代码充当强行终止按钮。

## 2. 它的机制和原理是什么？ (四大核心机制)

### 2.1 Token Budget 守护机制（成本阀门）
*   **机制原理**：为每一个任务预先分配一个“Token 预算”（就像充值卡）。大模型每次思考都会扣费。
*   **作用**：
    *   **70% 成本保护告警**：消耗达到 70% 时，系统会强制大模型降级（换成便宜的小模型）或缩小任务范围。
    *   **95% 强制熔断**：如果大模型陷入死循环疯狂瞎猜，达到预算 95% 时，不管任务完没完成，Harness 层都会直接“拉闸停电”，防止出现天价账单故障。

### 2.2 工具拦截器 (Tool Interceptor - 薄如蝉翼的旁路设计)
*   **机制原理**：它采取了“同步极简校验 + 异步旁路拦截”的设计。意思是，在主流程上只做最简单的检查（不浪费时间），把复杂的敏感词扫描等沉重工作扔到系统后台偷偷做。
*   **绝对核心防线：身份强制覆写**。不管大模型在调用工具时里面填了什么`租户ID`和`操作人`，拦截器都会冷酷无情地用**当前真实登录用户的信息**强行把它盖掉。这样就算大模型被黑客用“提示词注入”蛊惑去查别人的账，也绝对无法突破隔离。

### 2.3 Scratchpad 分片压缩审计 (防爆内存与黑匣子记录)
*   **机制原理**：大模型的每一步“心理活动（推理过程）”都被称为 Scratchpad。由于思考过程可能极其冗长，Harness 会把它们“切片”存在 Redis 里，并采用**滚动压缩**（比如只保留最近20步的完整话语，之前的全部压缩成一行“指纹”）。
*   **合规落库**：一旦大模型思考中触发了“转账”、“审批”等敏感高风险词汇，这一步的详细思考记录会被自动复印一份，永久存入 ClickHouse（一种分析型数据仓库）成为不可篡改的“呈堂证供”，方便以后人工查账和追责。

### 2.4 全局工具锁注册表 (死锁与DDOS并发防护)
*   **机制原理**：用 Redis 分布式锁，强制给各种工具设置了“并发上限”。
*   **作用**：很多银行老系统或企业内部旧系统非常脆弱。如果大模型的 Agent 起了多线程（并发），瞬间疯狂请求老系统，极可能把它彻底打挂掉（类似于自己攻击自己）。全局工具锁像水坝一样，卡死了最大并发量（比如遗留 MySQL 连接一次只放行 3 个）。

## 3. 还有其他什么可选技术方案？
*   **方案 A：大模型自我反思 (Self-Correction/Critique)**
    *   原理：在提示词里告诉大模型“你如果出错了，请自己检查并重试”。
    *   缺点：大模型陷入“钻牛角尖”时，它的反思反而会成倍消耗 Token，并不断重复同样的错。不能只靠它自己。
*   **方案 B：传统的 API 网关 (如 Nginx / Kong)**
    *   原理：在网关层通过正则匹配把危险请求拦下。
    *   缺点：太死板。难以动态追踪 Token 额度这种需要上下文的东西，而且无法处理大模型独有的“推理分片”和复杂的“工具 JSON Schema”。

## 总结
Harness 层就像是给一只聪明但危险的野马套上的“缰绳 (Harness)”。
它并不干涉大模型去如何聪明地进行推理和完成任务，但只要大模型这匹脚踏出**预算红线**、**租户数据特权红线**或**请求频率红线**，这根缰绳就会被立刻勒死。在企业级平台，这种“基于传统代码的生杀大权”不可转让给大模型自己。
