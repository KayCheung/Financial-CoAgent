一、总体评价（速览）
- 优点：已有清晰的控制塔（Harness）、规划层（LangGraph）、执行与沙箱、三层记忆（L1 Redis / L2 PG / L3 Milvus）、模型能力准入与 TokenBudget、工具拦截与审计轨道，满足金融私有化与合规需求的核心要素。很多扩展直接可复用这些基础模块。
- 需要补强的方面：实时协同/长连接能力、任务调度/持久化的专门调度器、细粒度协作权限模型、面向用户的工作流/看板前端契约、长期 persona 管理与多模态支持策略。

二、功能逐项可行性与设计建议

1) AI 分身（persona / AI 虚拟分身、长期角色）
- 可行性：高，但工程量中等到高（取决于分身的状态长度、多模态与长期运行能力）。
- 集成点：L3 语义记忆（Milvus）用于持久化分身记忆；Skill 中心用于封装分身能调用的能力集合；Harness 的能力注册表和 TokenBudget 用于管理分身的模型准入和成本；Scratchpad/L2 用于会话轨迹与审计。
- 必要改动/新增：
  - 分身管理服务（Persona Service）：CRUD、版本、能力标签、默认技能集、租户/用户绑定、权限与审计策略。
  - 长期会话/上下文窗口策略：如何 hydrate/dehydrate 分身（在 L1/L3 间做流式加载）。
  - Persona sandbox 与限制（避免擅自调用敏感工具）。
  - 多模态支持（头像、合成语音、角色图片）如需要，需引入额外模型与安全审计。
- 风险与缓解：
  - 泄露敏感记忆：对 L3 向量加密、租户隔离、访问审计、DLP 检测。
  - 成本膨胀：TokenBudget 扩展到 persona 级别配额、缓存策略、冷/热分身区分。
- 难度/优先级：中高；推荐作为中期功能（在看板/计划基本上线后并行开发）。

2) 类似 Jira 的看板（Kanban）
- 可行性：高，属于前端/业务层集成，对后端有明确需求但不复杂。
- 集成点：把 “线程/任务/子任务/DAG 节点” 映射为看板卡片。Scratchpad 的 step 与 L2 事件可作为卡片的历史与审计。Planner 生成的 DAG 与执行层状态用于卡片状态更新。Harness 的 ABAC 用于看板权限。
- 必要改动/新增：
  - 任务/看板服务（持久化 PG 表：board, column, card, card->thread 映射，watchers, comments）。
  - 实时推送（WebSocket / GraphQL subscriptions / SSE）用于前端实时呈现执行状态、审计事件、HITL 卡片。
  - UI 层实现与导出/报表接口。
  - 同步映射规则：如何将 Planner 节点、Executor 状态与卡片状态双向映射（建议事件驱动）。
- 风险与缓解：
  - 竞态与一致性：利用 L2 事件源（Event-sourced diffs）实现乐观并发或乐观锁，或使用 CRDT 做协作编辑（但复杂度高）。
  - 权限控制：复用 Harness 的 ABAC 与 ToolInterceptor 进行强制覆写。
- 难度/优先级：低到中；推荐作为第一批产品化功能（快速价值、可视化运维）。

3) 金融尽调 Plan 模式（标准化尽调流程/Checklist/模板）
- 可行性：高，与现有规划层/Skill/记忆层高度契合。
- 集成点：Planning 层用 LangGraph 生成尽调 DAG；Skill 中心封装尽调相关能力（OCR、对账、法人信息查询、RAG 检索、表格解析）；L2 保存审计与步骤；L3 用于检索历史尽调案例与相似风险点（RAG 支持）。
- 必要改动/新增：
  - Plan 模块模板引擎（模板 + 问卷 + 任务节点），支持版本化与审批门槛（发布准入）。
  - 流程引擎扩展（定制节点类型：文档收集、证据核验、外部数据抓取、审批节点）。
  - 尽调评分模型与 Critic 规则集（domain-specific rules）。
  - 输出标准报表与合规存档（ClickHouse/CH 已用于审计，可扩展 schema）。
- 风险与缓解：
  - 业务复杂度高、流程多变：提供可配置模板与低代码流程编辑。
  - 对外部数据依赖：ToolLock & retry 策略，明确 SLA 与超时/补偿事务（HITL 方案）。
- 难度/优先级：中；对金融场景价值高，建议优先级中高。

4) 多任务定时任务（scheduler / cron / 周期任务 / 多任务并发）
- 可行性：高，需引入专用调度或编排系统。
- 集成点：Planner/Executor 可作为被调度的工作单元；TokenBudget/Harness 管控每次触发的预算与权限；GlobalToolLock 管理资源并发。
- 必要改动/新增：
  - 持久化调度器（可选技术：Temporal、Airflow、Argo Workflows 或自建基于 Redis/PG 的 Scheduler + Worker）。建议使用 Temporal/Argo/Quartz-like 系统以确保可恢复性与可观测性。
  - 调度配置 UI（cron 表达式、依赖关系、并发限制、backfill）。
  - 长任务保活与续租（已在 ToolLockContext/heartbeat 有参考实现）。
  - 失败补偿与重试策略（结合 HITL 三段式审批与 WAL）。
- 风险与缓解：
  - 定时任务规模激增带来的并发与成本：将 TokenBudget 与全局并发配额结合，做批次化与速率限制。
  - 状态一致性：用事件溯源与快照确保调度器崩溃后恢复。
- 难度/优先级：低到中；工程较成熟，建议尽早引入成熟调度框架。

5) Web Search（实时网络检索 / 浏览器抓取 / 实时 RAG）
- 可行性：中等；技术可行但风险（法律/合规/PII/稳定性）与成本需控制。
- 集成点：作为一个 Tool（search_tool / browser_tool）接入 Skill 中心并受 Harness 拦截与 ToolLock 控制；检索结果可进 L3 向量化并参与 RAG；审计轨写入 ClickHouse。
- 必要改动/新增：
  - Web Retrieval Tool：支持第三方搜索 API（Bing/Google/SerpAPI）、或 headless 浏览器（Playwright）在受控沙箱内运行。
  - 缓存层与可信度打分（避免频繁请求与幻觉）；对检索结果进行来源标注与可追溯链接。
  - DLP/法律审查流程（禁止抓取受限站点、合规白名单/黑名单）。
  - Reranker（BGE）与可信来源优先策略。
- 风险与缓解：
  - 法律/版权/隐私：制定白名单、速率限制、日志留证、以及律师合规评估。
  - 数据漂移与不可靠性：结合多源比对、来源权重与 HITL 审核。
  - 成本与延迟：使用缓存、分级检索、并行召回降低延迟。
- 难度/优先级：中；若业务强依赖实时信息则优先，否则可作为增强项分阶段落地。

6) 协同（多人共同编辑、评论、审批、通知、版本与冲突解决）
- 可行性：中到高；涉及实时功能和复杂权限模型，需与审计与安全紧密结合。
- 集成点：看板服务、Plan 模块、HITL 审批（飞书卡片），L2 事件日志作为单一事实源支持冲突回放，Harness ABAC 控制权限。
- 必要改动/新增：
  - 协作模型：实时（CRDT 或 OT）vs 基于事件的乐观并发（更简单，适合多数业务）。
  - 权限与审计：扩展 ABAC、审计轨（谁修改了什么、何时、证据）。
  - 通知/审批桥接（支持飞书、邮件、Slack、企业微信）。
  - 共享线程 / 可见性策略与租户/项目隔离。
- 风险与缓解：
  - 实时一致性复杂：对于结构化任务对象建议先用事件驱动与乐观锁；对文档编辑若追求强实时体验再引入 CRDT。
  - 安全边界：复用 Harness 的强制覆写策略和工具拦截器。
- 难度/优先级：中高；按需分层实现（先做评论/审核/共享，再做强实时协作）。

三、跨功能的综合建议（架构级）
- 复用现有模块：
  - TokenBudget、能力注册表、ToolInterceptor、Scratchpad、GlobalToolLock、HITL 等均应作为新功能的第一级守护与接入契约。
- 事件驱动为首选整合方式：
  - 所有状态变更（看板操作、调度触发、Persona 更新、检索结果）应产生日志事件并入 L2（PG）或 Kafka，用于回放、审计与前端实时订阅。
- 权限与租户隔离：
  - 把看板/协作/分身的访问控制纳入 ABAC/能力注册表，细粒度到 action-level（创建、编辑、调用工具）。
- 长期成本控制：
  - 对“长期运行的分身/定时任务/频繁 web search”建立独立预算维度并在 TokenBudgetGuard 扩展到实体级（persona_id / schedule_id）。
- 测试与回放：
  - 新功能上线前建立 Evaluation 回放（文档已有发布准入原则），自动化压力/混合攻击测试、合规审查测试。
- 日志/可观测：
  - 前端行为、模型调用、工具调用、调度事件都要纳入 OTel/Langfuse/Grafana 的追踪/指标与尾部采样策略。

四、开发与落地路线（建议分阶段）
Phase 0（1-2 个月，快速上线价值）：
- 看板基础（后端Card/Board服务 + 前端最小 UI + 事件驱动同步 Exec 状态）。
- 持久化调度器的最简实现（基于 PG/Redis，支持 cron triggers 与 DAG trigger）。
Phase 1（2-3 个月）：
- 金融尽调 Plan 模式（模板、Planner 集成、报告输出）。
- 调度器健壮化（引入 Temporal 或 Argo，失败补偿、重试策略）。
Phase 2（3-4 个月）：
- 协同能力（评论、共享、审批流、通知）。
- Web Search 基础（受控检索工具 + 缓存 + RAG 集成）。
Phase 3（3-6 个月）：
- AI 分身（Persona 服务、L3 管理、成本管理、多模态扩展）。
- 强实时协作（若需要，CRDT/OT）。
总时间（分阶段并行团队）大致 4–9 个月可交付核心功能；AI 分身与强实时协作为耗时最大项。

五、风险清单（高优先级）
- 合规/法律（Web 搜索与外部抓取、数据留存策略、跨地域数据合规）。
- 成本飙升（长期运行分身、大规模检索、频繁模型调用）——需在设计期加强预算/配额、冷/热分离。
- 安全事故（工具越权、数据泄露）——Harness 层要强制执行覆写、审计与 DLP。
- 一致性/死锁/资源争用——GlobalToolLock、Planner 的防死锁 DAG、WAL 补偿机制是关键；需要压力测试覆盖常见争用场景。

六、具体实现注意点（工程细节）
- 将 persona/plan/board 都视作“元对象”，统一元数据与事件模型，便于权限与审计复用。
- Scheduler 选型建议：Temporal（长期任务/状态持久化强）、Argo（K8s 原生）、若轻量可自建基于 Redis streams + workers。
- Web search tool 必须走沙箱（gVisor/runc）并纳入 ToolLock，且返回结果需带来源与可信度分数。
- 协作同步首选事件驱动 + WebSocket；对文档编辑短期使用乐观并发与版本历史。
- 扩展 TokenBudget 支持更多维度（entity-level, periodic quotas），并在能力注册表中把每个 Skill/Tool 标注成本估算。

七、总结性的建议（行动项）
1. 立即实现看板 + 调度基础，作为最小可用产品（快速获得用户反馈）。  
2. 在实现任何长期/外部数据功能前，先补齐 Persona/Plan 的权限模型与审计策略。  
3. 将 TokenBudget 扩展到实体级（persona、schedule），并建立成本告警与自动降级策略。  
4. Web Search、AI 分身、强协同作为后续增强，优先做合规与沙箱验证。  
5. 制定分阶段测试矩阵：安全、合规、成本、恢复性、并发/死锁场景。
