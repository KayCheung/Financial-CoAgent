# 企业级 Agent 平台分阶段实施方案

> **基准文档**：基于 DeerFlow 的企业级 Agent 平台完整方案 v1.6  
> **编制日期**：2026-04-08  
> **核心原则**：每阶段可独立交付成果 · 任务间低耦合/接口化依赖 · AI 辅助编码加速

---

## 总览：四阶段里程碑

| 阶段 | 周期 | 核心目标 | 可见成果 |
|------|------|----------|----------|
| **一期：基座闭环** | M1-M2（8周） | 跑通对话 + OCR 发票全链路 | 用户可正常对话、上传发票自动识别并生成凭证 |
| **二期：Harness 构建** | M3-M4（8周） | 安全护栏 + 记忆体系 + HITL | 工具拦截生效、审批流程可用、多轮上下文连贯 |
| **三期：业务深水区** | M5-M6（8周） | 脱敏网关 + MCP 问数 + 财务Agent | 财务报销自动闭环、审批卡片、PII 脱敏可用 |
| **四期：平台化运营** | M7-M10（16周） | Skill 广场 + 可观测 + 生产就绪 | 全链路监控、发布门禁、压测通过、全面投产 |

## DeerFlow 基座评审前提

本方案默认基于 DeerFlow 2.0 + LangGraph 1.0 的工作流底座。以下内容在进入各阶段实施前必须先确认：

- DeerFlow 在本项目中只承担工作流编排内核，不默认包含全部业务能力。
- `Tool`、`Skill`、`Agent` 的职责边界要统一，不能在不同任务里各自解释。
- `thread_id`、`trace_id`、`tenant_id` 等上下文字段要贯穿网关、引擎、工具、审计层。
- 状态持久化优先采用 Redis Checkpointer + PostgreSQL 审计，恢复和回放策略先定。
- 模型接入层默认只做统一网关与熔断，不在首期引入复杂 prompt 平台。
- 涉及跨租户、跨权限、跨审批的能力，默认必须经过网关和审计，不允许旁路调用。

### 架构收敛声明对齐（v1.6 §1.2.1）

本实施方案遵循 v1.6 §1.2.1 定义的"少数硬约束优先"建设原则，实施范围按以下三类硬约束组织：

| 硬约束类别 | 覆盖任务 | 实施阶段 |
|-----------|---------|----------|
| **下游系统边界**：租户隔离、权限控制、Tool Schema 约束 | T1.1（网关）、T1.11（PG RLS）、T2.1（工具拦截器）、T3.5（ABAC） | 一期 ~ 三期 |
| **状态与审计可恢复**：WAL、快照检查点、失败态分类 | T1.8（WAL 审计）、T2.2（Scratchpad）、T2.4（三层记忆） | 一期 ~ 二期 |
| **发布准入**：Evaluation 回放、基线门禁 | T4.4（Evaluation Layer）、T4.1（Skill 上架审核） | 四期 |

其余能力（强沙箱、影子验证、OCR 升维等）作为**风险增强项**，依据业务风险等级按需启用。

## 第一阶段：基座闭环（M1-M2，8周）

### 阶段目标

**让用户看到可交互的成果**：能正常对话问答、能上传发票图片完成 OCR 识别并输出结构化结果。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 对话可用性 | 用户通过 Web/API 发送自然语言问题，Agent 在 10s 内返回有效回复 |
| OCR 发票识别 | 上传标准增值税发票图片，关键字段（发票号、金额、税号、日期）提取准确率 ≥ 90% |
| 路由命中率 | Semantic Router 在冷启动配置下，意图分类准确率 ≥ 70% |
| 系统稳定性 | 单用户连续对话 50 轮无崩溃，API 可用率 ≥ 99% |
| Token 成本可观测 | 每次对话的 Token 消耗可在日志中查看 |

### 所需中间件

| 中间件 | 版本 | 用途 | 部署要求 | 最低资源 |
|--------|------|------|----------|----------|
| PostgreSQL | 15+ | 业务数据、事件日志、审批记录 | 单实例（规格A），开启 WAL | 2C4G, 50GB SSD |
| Redis | 7.0+ | 会话状态、Token 计数、SR 缓存 | 单节点（规格A） | 1C2G, 10GB |
| Milvus | 2.3+ | Semantic Router 向量检索 | Standalone 模式 | 2C4G, 50GB SSD |
| Kong/APISIX | 3.6+/3.0+ | API 网关，限流/JWT/路由 | 单节点 | 1C2G |
| PaddleOCR | PP-Structure | 发票版面解析 | Docker/GPU 可选 | 2C4G（CPU）或 1GPU |
| Nacos | 2.3+ | 配置中心（热更新） | 单节点 | 1C2G |
| Python Runtime | 3.11+ | DeerFlow + LangGraph + LangChain | 容器化部署 | 4C8G × 2 |
| LLM API | Doubao/DeepSeek/Qwen | 主力模型 API 访问 | API Key 配置 | 按调用量计费 |

> **总计最低资源**：约 16C 30G + 100GB SSD（不含 GPU）

### 任务拆分

#### T1.1 企业安全网关搭建

**类型**：独立任务（无前置依赖）  
**预估工时**：3天

**范围**：
- 部署 Kong/APISIX 网关，配置 JWT 验签、限流、CORS
- 配置统一入口路由规则（/api/v1/chat, /api/v1/ocr 等）
- 实现租户上下文注入（从 JWT 解析 tenant_id、user_id、role 写入请求头）

**细节补充**：
- JWT 密钥管理：首期使用对称密钥（HS256），密钥存 Nacos，后续升级 RS256
- 限流策略：按租户 100 req/min，按 IP 200 req/min，防突发流量
- 健康检查端点：/health 不鉴权，供 K8s 探针使用
- 错误响应格式：统一 JSON 结构 `{ code, message, trace_id }`
- SSL/TLS：首期内网 HTTP，生产前必须配置 HTTPS

**验收标准**：通过 JWT Token 可访问受保护 API，无 Token 返回 401

#### T1.2 DeerFlow 核心引擎部署

**类型**：独立任务  
**预估工时**：5天

**范围**：
- 搭建 DeerFlow 2.0 + LangGraph 1.0 + LangChain 0.3 项目骨架
- 实现 `AgentFastState` 状态定义（精简版，仅含核心字段）
- 实现基础 LangGraph 工作流：`router → planner → executor → END`
- 接入 LLM 统一网关（支持 Doubao/DeepSeek 切换）

**细节补充**：
- 项目结构建议：`src/core/`（引擎）、`src/harness/`（护栏）、`src/skills/`（技能）、`src/tools/`（工具）
- LLM 网关首期简化：只做模型切换 + 超时熔断，不含完整能力注册表
- 状态持久化：首期使用 Redis Checkpointer，key 格式 `checkpoint:{thread_id}`
- 环境变量管理：模型 API Key、数据库连接串通过 Nacos 统一管理
- 日志规范：结构化 JSON 日志，必须包含 `thread_id`、`tenant_id`、`trace_id`
- 错误处理：所有节点异常必须映射到 `AgentFailureState` 三分类

**验收标准**：发送自然语言问题到 /api/v1/chat，能获得有效的 LLM 回复

#### T1.3 Semantic Router 基础版

**类型**：依赖 T1.2（接口依赖——需要 LangGraph 定义的 router 节点接口）  
**预估工时**：5天

**范围**：
- 部署 Milvus 向量库，创建意图向量 Collection
- 使用 `bge-small-zh-v1.5` 进行 query embedding
- 实现冷启动配置（阈值下调至 0.75，抽样率 20%）
- 实现 LLM fallback 兜底路由

**细节补充**：
- 向量维度：bge-small-zh-v1.5 输出 512 维，Milvus 配置 COSINE 度量
- 初始意图集：至少预置 `general_chat`、`invoice_ocr`、`knowledge_qa` 三个意图
- 每个意图至少准备 20 条种子样本作为向量库初始数据
- 冷启动退出条件：全局请求数 ≥ 1000 且最近 200 次影子验证不一致率 ≤ 15%
- 影子验证日志：写入 PG 表 `router_shadow_log`，供后续分析
- Milvus Collection Schema：`{id, intent, embedding, tenant_id, created_at}`
- **注意**：首期不实现二级路由和自动微调，仅做基础向量匹配 + LLM 兜底

**验收标准**：不同类型问题（闲聊/发票/知识查询）能正确路由到对应 Skill

#### T1.4 Token Budget 守护机制

**类型**：接口依赖 T1.2（需要 T1.2 完成 LLM 调用 wrapper 接口后集成）  
**预估工时**：3天

**范围**：
- 实现 `TokenBudget` 预算分配（simple/moderate/complex 三档）
- 实现 `TokenBudgetGuard`：Redis 幂等计费 + 70%降级告警 + 95%硬熔断
- 集成到 LangGraph 工作流中（每次 LLM 调用前后扣减 Token）

**细节补充**：
- Redis Lua 脚本：幂等计费脚本需要在部署时预加载（SCRIPT LOAD）以减少网络开销
- Redis 降级：当 Redis 不可用时降级到进程内计数，但必须打告警日志
- 指标上报：首期写入结构化日志，不强制对接 Prometheus/Grafana
- TTL 设计：预算 key 24h 过期，operation_id 集合同步过期
- 并发安全：多 Pod 部署时依赖 Redis 原子操作保证计数一致，本地计数仅临时容错
- 与 LLM 网关集成：在 LLM 调用 wrapper 中自动调用 `guard.consume()`
- **Redis 恢复回填策略（C8）**：
  - **触发条件**：Redis 恢复可用后，由 `TokenBudgetGuard` 在下次 `consume()` 调用时检测到 Redis 可达，自动触发回填
  - **回填来源**：从 PG 审计表 `agent_audit_log` 中按 `thread_id` 聚合该线程的历史 Token 消耗总量，写回 Redis counter key
  - **一致性校验**：回填后比对 Redis 值与 PG 聚合值，偏差 > 5% 时打 WARN 日志并以 PG 值为准
  - **防重复回填**：使用 Redis `SETNX` 加回填锁（key: `token_budget:{tenant_id}:{thread_id}:backfill_lock`，TTL 60s），保证多 Pod 场景下只有一个 Pod 执行回填
  - **降级期间的预算保护**：本地计数期间，预算硬熔断阈值从 95% 下调至 85%，防止 Redis 不可达期间因本地计数不精确导致超支

**验收标准**：超预算请求被明确拦截并返回错误；Redis 故障恢复后预算数据可自动回填且偏差 ≤ 5%

#### T1.5 Planner 防死锁机制

**类型**：依赖 T1.2（接口依赖需要 LangGraph planner 节点）  
**预估工时**：3天

**范围**：
- 实现 DAG 环路检测（拓扑排序 DFS）
- 实现 Planner 自纠错：重试上限 2 次 + 携带错误上下文
- 实现 Critic 后路由：超时熔断（30min）+ 重试上限 3 次

**细节补充**：
- DAG 校验三件事：环路检测 + 依赖完整性校验 + 孤立节点检测
- 空 DAG 处理：Planner 输出空计划视为结构非法，计入 `plan_retry_count`
- 错误上下文格式：`last_planner_error` 必须是结构化字符串，告诉模型修正方向
- 幂等键设计：Planner/Executor 每个节点需生成唯一 `idempotency_key`
- 超时计时：从 `state.start_time` 开始，30 分钟全局硬熔断
- **Critic 首期简化**：首期 Critic 只做格式校验（输出是否符合 Schema），不做复杂业务校验

**验收标准**：生成含环路的 DAG 时被拦截；Planner 错误后能携带上下文自动修正

#### T1.6 PaddleOCR 发票解析服务

**类型**：独立任务  
**预估工时**：5天

**范围**：
- 部署 PaddleOCR PP-Structure 服务（Docker 容器化）
- 实现发票 OCR 统一入口 API（支持图片上传  结构化字段提取）
- 实现 Claim-Check 模式：图片  OSS 存储  返回 `doc_uri`
- 将 OCR 封装为 LangGraph 可调用的 Tool

**细节补充**：
- 支持格式：JPG/PNG/PDF，单文件  20MB
- 输出字段：发票号码、发票代码、开票日期、金额（不含税）、税额、价税合计、销方名称/纳税人识别号、购方名称/纳税人识别号
- 置信度阈值：单字段置信度 < 0.7 时标记为 `low_confidence`，提醒人工复核
- 首期不启用 PaddleOCR-VL 局部升维（留接口，二期实现）
- Claim-Check 实现：使用对象存储适配层（首期可用本地文件系统模拟 OSS）
- 异步处理：首期同步调用（单张发票 < 10s），不启用 Suspend & Resume
- Tool Schema 定义：`{ name: "invoice_ocr", params: { doc_uri: str }, returns: { fields: dict, confidence: dict } }`
- 错误处理：图片损坏/格式不支持返回 `HUMAN_REVIEW` 级别错误

**验收标准**：上传增值税发票图片，返回结构化字段 JSON，关键字段准确率  90%

#### T1.7 知识库 Agent（第一个 Skill）

**类型**：依赖 T1.3（接口依赖需要 Router 路由到该 Skill）  
**预估工时**：5天

**范围**：
- 实现基础 RAG 链路：文档上传  切片  Embedding  Milvus 存储
- 实现知识检索 Skill：用户提问  向量召回  Reranker  LLM 生成回答
- 租户隔离：向量检索强制注入 `tenant_id` 过滤

**细节补充**：
- 文档切片策略：按段落切片，单片  512 tokens，重叠 50 tokens
- Embedding 模型：复用 bge-small-zh-v1.5
- Reranker：首期使用 bge-reranker-base，Top-K=5 召回后 Rerank 取 Top-3
- Milvus Collection：`skills_knowledge`，Schema 包含 `{doc_id, chunk_text, embedding, tenant_id, skill_id}`
- 文档格式支持：首期支持 txt/md/pdf，后续扩展 docx/xlsx
- 无结果处理：检索无结果时，返回 "未找到相关信息，建议联系人工客服"
- 引用溯源：回复中附上引用的文档片段 ID 和相关度分数
- **Milvus 多租户隔离策略与 ANN Recall 权衡（v1.6 §4.4.3）**：
  - **策略选择**：v1.6 已将"金融核心租户"的默认策略升级为 **Skill 维度独立 Collection**，而非共享 Collection 加 `tenant_id` 过滤；多中小租户（无金融核心业务）允许使用共享 Collection，但必须加 `partition` 分区辅助隔离，不得以 `expr` 过滤作为唯一防线
  - **共享 Collection 的 Recall 影响**：当使用 `expr: tenant_id == "xxx"` 过滤时，Milvus IVF_FLAT/IVF_SQ8/HNSW 索引的实际召回率相较于全量 ANN 搜索**通常下降 5%-20%**，具体取决于该租户数据占总向量库的比例（占比越低，过滤比例越高，recall 退化越严重）；当目标租户数据量 < 全库 10% 时，HNSW 索引在 top-10 召回场景下 recall@10 可能跌至 0.65-0.75（vs 无过滤时的 0.92+）
  - **业务可接受阈值**：本项目设定最低 **recall@5 ≥ 0.85**（结合 Reranker 后的最终 top-3 准确率 ≥ 80%）为验收红线；若线上监控发现 recall 低于该阈值，则强制从共享 Collection 迁移到独立 Collection
  - **首期实施原则**：一期以单租户或少量租户为主，暂用共享 Collection + `tenant_id` 过滤，部署后通过影子查询评估 recall；当任意租户数据量占全库 ≤ 15% 时，自动告警并评估迁移独立 Collection 的必要性
  - **监控指标**：需在 T4.5 可观测中加入向量检索 `recall@5_estimated` 指标（基于 ground-truth 测试集离线评估）和 `nq_filter_ratio`（每次搜索中被过滤掉的向量比例）

**验收标准**：上传知识库文档后，用户提问能获得基于文档内容的准确回答

#### T1.8 WAL 双写审计基础版

**类型**：独立任务  
**预估工时**：3天

**范围**：
- 实现 `LocalWALStore`：本地文件 WAL（fsync 保证落盘）
- 实现 `secure_audit_flow`：先写本地 WAL，异步写 PG 热审计表并发送阿里云 ONS（RocketMQ 版）事件流
- 实现决策证据链记录：工具调用、参数快照、结果快照

**细节补充**：
- WAL 目录：`/var/agent/wal/`，需保证目录权限和磁盘空间
- WAL 文件格式：`{entry_id}.json`，fsync 强制落盘
- 一期接入阿里云 ONS（RocketMQ 版）审计事件总线；ONS/RocketMQ 不可用时，WAL 保留待 Janitor 重试，保证审计事件不丢
- PG 表 `agent_audit_log` 作为热审计查询与 Evaluation 回放源，表结构：`{ id, thread_id, tenant_id, entry_type, content_json, created_at }`
- Janitor 首期简化：不单独部署 DaemonSet，用定时任务扫描 WAL 目录重试
- 审计内容：记录决策摘要（非完整思维链）、工具调用参数、执行结果、规则命中情况
- 日志保留：PG 中保留 90 天，超期归档（首期手动归档）
- **Evaluation 格式对齐（C6）**：WAL 条目的 `content_json` 字段结构应与四期 Evaluation Layer 的 replay 格式预先对齐，建议字段包含 `{ input, router_result, plan, tool_calls, output, latency_ms, token_consumed }`；此约定在一期写死，避免四期改造审计历史数据；设计时在 WAL 写入层预留 `schema_version` 字段（首期值 `"v1"`）

**验收标准**：每次 Agent 执行都有审计记录，WAL 文件在成功同步后自动清理

#### T1.10 LLM 能力注册表 MVP

**类型**：接口依赖 T1.2（需要 T1.2 完成模型接入层骨架后注册）  
**预估工时**：3天

**背景**：设计方案 v1.5 第 1.4 节明确规定"统一的是调用协议，不统一能力承诺"，模型降级仅允许发生在**能力兼容**前提下。若无能力注册表，金融场景下 Planner/Critic 节点可能在降级时被切换为不满足结构化输出要求的模型，造成计划格式漂移或审批逻辑失效。

**范围**：
- 实现 `ModelCapabilityRegistry`：存储每个模型版本的能力标签和禁止使用节点
- 实现节点准入检查：在 Planner/Critic 节点调用前，校验当前模型是否满足最低能力要求
- 实现兼容性降级：降级前先查注册表，无合法替代时允许缩小任务范围或转 HITL，不允许无约束降级

**能力标签定义（首期实现）**：

| 标签 | 含义 | 适用节点 |
|------|------|----------|
| `structured_output_strict` | 严格输出 JSON/Schema，不漂移字段 | Planner、Critic、状态抽取 |
| `tool_calling_stable` | 工具调用服从性稳定，低幻觉参数 | Planner、Executor |
| `finance_chinese_reasoning` | 中文金融术语、票据、审批语境理解稳定 | Critic、报告生成 |
| `low_latency_router` | 低延迟分类与路由稳定性 | 路由兜底分类 |

**节点最低准入矩阵（硬约束）**：

| 节点 | 最低能力要求 | 不满足时策略 |
|------|------------|--------------|
| Planner | `structured_output_strict` + `tool_calling_stable` | 缩小任务范围或显式失败，不假降级 |
| Critic | `structured_output_strict` + `finance_chinese_reasoning` | 回退规则引擎 + 转 HITL |
| 路由兜底 | `low_latency_router` | 路由到默认安全 Skill |

**细节补充**：
- 注册表存储：首期使用 Nacos 配置（JSON 格式），支持热更新，格式 `{ model_id, version, capabilities: [], forbidden_nodes: [] }`
- 准入检查时机：在 LLM 调用 wrapper 的 `_select_model(node_type)` 方法中执行，不在业务节点内部处理
- 降级日志：每次发生能力兼容性降级必须打 WARN 级别日志，包含 `from_model`、`to_model`、`node_type`、`reason` 字段
- 首期不做自动化能力评测，能力标签由人工在模型注册时填写并经验证后生效
- 与 T1.2 集成：T1.2 的 LLM 网关 wrapper 需预留 `get_model_for_node(node_type)` 接口，T1.10 实现具体路由逻辑
- 不满足准入时的失败模式：返回 `FAILED_CLOSED` 分类错误，不允许静默降级

**验收标准**：为 Planner 节点指定不满足 `structured_output_strict` 的模型时，系统拒绝并返回明确错误；降级路径有日志可查

#### T1.11 PostgreSQL 多租户 RLS 配置

**类型**：依赖 T1.2（接口依赖——需要 T1.2 完成 PG 表结构定义后配置 RLS 策略）  
**预估工时**：2天

**背景**：v1.6 §4.4.2 明确要求生产环境默认开启 PostgreSQL `ROW LEVEL SECURITY`。租户隔离不能仅依赖应用层 `tenant_id` 注入，必须在存储层形成硬隔离兜底。

**范围**：
- 对核心业务表（`agent_memory`、`agent_audit_log`、`hitl_resource_snapshot` 等）启用 RLS
- 实现 `tenant_isolation_policy`：`USING (tenant_id = current_setting('app.current_tenant')::text)`
- 在数据库连接池初始化后注入会话变量 `SET app.current_tenant` 和 `SET app.current_user`
- 审计专用角色走独立通道，不直接绕过 RLS

**细节补充**：
- RLS 策略覆盖表清单：`agent_memory`、`agent_audit_log`、`router_shadow_log`、`hitl_resource_snapshot`、`state_events`、`state_checkpoints`
- 连接池集成：在 `asyncpg` / `SQLAlchemy` 连接获取后、执行查询前，自动注入 `SET app.current_tenant = '{tenant_id}'`
- 后台管理查询：跨租户查询走专用 `audit_admin` 角色 + 独立连接池，查询日志必须写入审计表
- 首期不做列级加密，只做行级隔离
- 测试要求：构造跨租户查询场景，验证 RLS 策略生效（应返回空结果而非报错）

**验收标准**：租户 A 的数据库连接无法查询到租户 B 的数据；跨租户管理查询有审计记录

#### T2.5 HITL 三段式审批状态机

**类型**：独立任务（适配 LangGraph 节点接口）  
**预估工时**：5天

**范围**：
- 实现 `ApprovalRecord` + `ApprovalStatus` 状态模型
- 实现 `HITLApprovalStateMachine`：提交 → 等待 → 催办 → 超时否决
- 实现资源补偿事务（超时后释放预占 Pod/OSS 对象）
- 集成飞书/钉钉审批卡片

**细节补充**：
- 幂等提交：相同 thread_id 的重复提交返回原 approval_id
- CAS 状态更新：状态变更必须 `PENDING → target_status`，禁止无条件覆盖
- 超时差异化：normal 24h、high 4h、urgent 1h
- 催办时机：超时时间的 75% 时发送催办
- 等待机制：轮询 PG 持久态（Pod 重启可恢复），非进程内 Event
- 轮询退避：urgent 0.5s 起步/max 5s，normal 2s 起步/max 15s
- 并发上限：单 Pod 轮询信号量 200，防止压垮 PG
- 进程启动恢复：扫描 PENDING 审批写审计标记
- **IM 适配层**：飞书/钉钉封装为统一接口 `IMClient.send_approval_card()`
- **资源补偿事务完整机制（C3）**：
  - **快照存储**：审批提交时，将已消耗资源的引用快照（`{ pod_ids: [...], oss_object_keys: [...], db_locks: [...] }`）写入 PG 表 `hitl_resource_snapshot`，关联 `approval_id`；不存 Redis（保证 Pod 重启可恢复）
  - **补偿触发者**：由审批状态机自身在状态切换为 `TIMEOUT_REJECTED` 或 `CANCELLED` 时同步触发补偿，不依赖外部 Janitor
  - **补偿操作**：按顺序释放预占 Pod slot → 删除 OSS 临时对象 → 释放数据库行锁；每个操作幂等，任意失败后最多重试 3 次
  - **补偿失败处理**：3 次重试仍失败，则写入 `hitl_compensation_failed` 告警表并触发告警通知，不阻塞审批状态机继续流转
  - **已批准时跳过**：状态切换为 `APPROVED` 时直接删除快照记录，不执行补偿操作，资源继续被后续执行使用

- **11 项核心告警指标**：
  1. LLM 调用延迟 P99
  2. Token 预算超限率
  3. 工具调用失败率
  4. Planner 重试率
  5. HITL 审批超时率
  6. 跨租户数据泄露拦截次数
  7. WAL 同步失败率
  8. Redis Checkpointer 失败率
  9. Semantic Router 影子验证不一致率
  10. 向量检索 recall@5 估计值（离线评估）
  11. 整体请求错误率
- **Dashboard**：至少 3 个面板：系统概览、租户维度、异常详情

**验收标准**：核心 Span 覆盖率 100%；11 项告警规则全部配置并可触发

| 风险 | 影响阶段 | 应对措施 |
|------|----------|----------|

> **DeerFlow 6-8 周替代预案工作量依据**（v1.6 风险矩阵补充说明）：
>
> 本项目架构决策的核心前提之一是 **Harness 层完全外挂、不侵入 DeerFlow 核心代码**。因此"替换 DeerFlow"等价于"替换工作流编排内核"，而非重写整个平台。6-8 周的估算基于以下维度：
>
> | 工作项 | 估算（AI 辅助编码） | 说明 |
> |---|---|---|
> | LangGraph 直接对接 + StateGraph 重建 | 2 周 | 状态模型已有 `AgentFastState` 定义，重新绑定 LangGraph 节点 |
> | Harness 层适配（Tool Interceptor / Token Budget / WAL 接口对接）| 1.5 周 | Harness 接口化设计，仅需替换注入点 |
> | Planner/Critic/Executor 节点重写 | 1.5 周 | 节点逻辑独立，DAG 校验、重试均在本方案内实现 |
> | 三层记忆模型兼容适配 | 1 周 | L2/L3 数据结构已独立到 PG/Milvus，只需重写 hydrate/dehydrate 接口 |
> | 集成测试 + 验收 | 1 周 | 基于已有验收样例集回放 |
> | **合计** | **~7 周** | AI 辅助编码下可达成，纯人工则需 × 2-3 |
>
> **不包含**：Skill 中心底层重构、新评测集建立、生产压测、Milvus/Redis 数据迁移。若上述项均需重做，总工期升至 12-16 周。
>
> **触发条件**：DeerFlow 合规审查无法通过（如开源许可证冲突、监管要求私有化源码审计不允许引入特定依赖）时启动，替代目标为纯 LangGraph + 本方案 Harness 层。

## 补充说明一：使用 DeerFlow 作为基座时，必须先明确的内容

下面这些不是“可选实现细节”，而是基于 DeerFlow 作为底座时，必须在实施方案里先写清楚的基线。否则任务接单后，开发、测试、运维、验收都会出现理解偏差。

### 1. 基座边界必须明确 `[已解答: 见 T1.2 引擎部署]`

- DeerFlow 在本项目中到底是“完整业务框架”还是“工作流编排内核”。
- 哪些能力属于 DeerFlow 基座，哪些能力属于本方案新增模块，哪些能力只做适配层。
- 是否允许直接修改 DeerFlow 核心代码，还是只能通过扩展点、插件、适配器实现。
- DeerFlow 的版本号、分支策略、依赖锁定策略是否固定。
- 现有仓库中与 DeerFlow 相关的目录、包名、命名规范、构建方式是否统一。

### 2. 执行模型必须明确 `[已解答: 见 T1.2 / T1.5]`

- 任务执行是同步、异步、还是混合模式。
- `router -> planner -> executor` 是否是唯一主链路，还是允许旁路工具流。
- 每个节点的输入、输出、失败态、重试态、超时态是否统一。
- `thread_id`、`run_id`、`task_id`、`trace_id` 的含义和生命周期。
- 是否允许长任务跨进程恢复，恢复点保存在哪里。

### 3. 状态模型必须明确 `[已解答: 见 T1.2 / T2.4]`

- `AgentFastState` 的字段集合、不可变字段、可选字段、版本号演进规则。
- 哪些状态保存在内存，哪些状态落 Redis，哪些状态落 PostgreSQL。
- Scratchpad、短期记忆、长期记忆、审计日志之间的边界。
- 状态快照如何回放、如何对比、如何做兼容升级。
- 状态丢失后是重算、回滚还是人工介入。

### 4. 工具与 Skill 体系必须明确 `[已解答: 见 T1.7 / T4.1]`

- DeerFlow 里 `Tool`、`Skill`、`Agent` 的职责边界。
- Skill 的注册方式、元数据格式、目录结构、版本发布规则。
- 工具调用是否要统一经过拦截器和审计层。
- 工具 schema 的定义语言是什么，JSON Schema、Pydantic、还是自定义协议。
- Tool 的幂等性、重试、超时、返回体结构是否统一。

### 5. 模型接入必须明确 `[已解答: 见 T1.10 能力注册表]`

- 支持哪些模型供应商，是否要求可热切换。
- 统一网关是否只是“模型代理”，还是还承担 prompt 模板、路由、限流、熔断。
- prompt 模板、系统提示词、工具提示词的存放位置和版本控制方式。
- 失败回退规则：模型失败后是切换模型、降级规则引擎、还是直接失败。
- 流式输出、截断输出、超长上下文的处理方式。

### 6. 记忆体系必须明确 `[已解答: 见 T2.4 三层记忆模型]`

- DeerFlow 是否默认有 memory 层，如果有，哪一层是框架能力，哪一层是业务能力。
- L1/L2/L3 的读写顺序、合并规则、覆盖规则、TTL。
- 记忆检索是“先检索后生成”，还是“先生成后回写”。
- 记忆污染如何防止，特别是跨租户、跨任务、跨回合。
- 是否允许从历史对话中自动提炼知识并写入知识库。

### 7. 安全与租户隔离必须明确 `[已解答: 见 T1.1 / T1.11 / T3.5]`

- 网关是否是唯一可信入口，内部服务是否一律不信任客户端 Header。
- 租户信息是由 JWT、mTLS、还是服务间签名下发。
- 细粒度授权是 RBAC、ABAC，还是二者并存。
- 敏感字段脱敏、Tokenization、反脱敏、审计可见性如何分级。
- Prompt Injection、越权工具调用、跨租户召回如何防护。

### 8. 可观测性必须明确 `[已解答: 见 T4.5 全链路可观测]`

- 标准日志字段集合必须统一：`tenant_id`、`thread_id`、`trace_id`、`run_id`、`skill_id`、`tool_name`。
- 指标、日志、Trace、Eval 的归档规则是否统一。
- 哪些事件必须进入审计链，哪些事件只进业务日志。
- 是否需要回放能力，回放粒度到什么层级。
- 错误分类是否统一为 `RETRYABLE / HUMAN_REVIEW / FAILED_CLOSED / INTERNAL_ERROR` 之类。

### 9. 部署与运行时必须明确 `[已解答: 见 T4.6 生产规格扩展]`

- 是单体部署、模块化单体，还是按服务拆分。
- Docker 镜像是统一基座镜像，还是每个模块单独镜像。
- K8s 是否是唯一目标环境，是否支持本地开发和离线部署。
- Redis、PG、Milvus、OCR、LLM 网关是否允许本地模拟。
- 配置中心、密钥管理、证书轮换策略如何落地。

### 10. 测试与验收必须明确 `[已解答: 见 T4.4 / T4.7 / T4.8]`

- 单测、集成测试、回归测试、压测、红队测试分别由谁负责。
- 每个任务的验收是“接口可用”还是“业务样例通过”还是“性能指标达标”。
- 是否要保留黄金样例集、回放集、对抗集。
- 方案里的数值指标是否是硬门槛，还是阶段性目标。

- 多租户隔离是逻辑隔离优先，还是物理隔离优先。 `[已决: 方案 T1.11 定义了利用 PG RLS 行级硬隔离，优先逻辑硬隔离]`

## 补充说明五：面向二期的底层延展设计（前瞻性规范）

虽然本实施方案属于一期建设范畴，但为了保证二期核心功能（看板、调度器、并发尽调、协同）的平滑演进，避免未来出现“锁大表 DDL 修改”或从底层重构权限模型的情况，一期在代码实现时必须**强行遵守**以下前瞻性规范：

### 1. 数据库设计的“宽表与扩展字段”规范（为二期协同与调度器铺路）
- **实施强制要求**：在一期的所有核心业务表（尤其是 `agent_token_events` 和 `agent_threads` 表）中，**强制增加一个 `metadata` (数据类型：JSONB) 字段**。
- **目标**：为二期留出免 DDL 扩容区。未来的 `entity_id`、`entity_type`、协同分享权限组等字段统统塞入此处，平滑向二期过渡。

### 2. API Request Schema 的防腐预留（为二期调度器铺路）
- **实施强制要求**：在暴露给前端的 API 接口（如 `/api/v1/chat` 或 `TriggerLangGraphWorkflow`）的 Pydantic 模型（请求入参）中，必须显式预留一个 `trigger_source` 字段，一期后端默认将其设为 `manual`。
- **目标**：二期的调度器微服务上线后，可以直接复用并调用同一接口传入 `trigger_source=scheduler`，实现调度任务与人工操作在网关和审计日志中的统一识别分流。

### 3. RocketMQ 审计载荷的完备性（为二期看板铺路）
- **实施强制要求**：往 RocketMQ 推送 L2 Audit Log 事件流时，JSON Payload 中必须带有严格的 LangGraph 节点流转映射（如：`current_node: planner`, `next_node: executor`）及生成时间戳。
- **目标**：二期的看板服务只需纯异步订阅 RocketMQ，无需向一期 PostgreSQL 数据库发起大量反查，即可做到物理层解耦的 Kanban 卡片状态实时同步。

### 4. 接口层与应用层的“全量 Owner 意识”（为二期并发尽调铺路）
- **实施强制要求**：一期代码中所有的 SQL `WHERE` 条件除了查 `id` 外，必须带上 `tenant_id = ? AND owner_id = ?`（即便是单人单线程的表）。
- **目标**：为二期提供免费的并发与权限墙。二期的调度大脑可以在上层分配子任务给不同的 Owner 去并发执行，一期底层代码无需因为“协同”的加入而产生跨租户/跨用户越权。

### 5. 二期新表的 RLS 继承规范（为协同、看板、调度器、尽调铺路）
- **实施强制要求**：一期建表与 DAO 模板必须沉淀为通用规范：所有二期新增 PG 表默认包含 `tenant_id`，涉及用户可见性或写入归属的表默认包含 `owner_id` 或等价主体字段，并在迁移脚本中同步开启 RLS。
- **目标**：二期的 `thread_shares`、`kanban_boards`、`kanban_columns`、`kanban_cards`、`card_watchers`、`dd_plan`、`scheduler_jobs` 等表具备统一的租户隔离、归属校验与子表访问保护能力。

### 6. Web Search 审计字段预留（为二期独立搜索入口铺路）
- **实施强制要求**：一期 WAL/PG 审计表的 `content_json` 或 `metadata` 中预留 `query_hash`、`redacted_query`、`pii_tags`、`source_domain`、`credibility` 等通用字段，原始搜索词或外部检索内容不得默认明文进入普通审计轨。
- **目标**：二期 Web Search 既能作为尽调内部 Tool，也能作为独立对话入口，同时满足金融场景下的敏感查询脱敏、引用溯源和合规审计要求。
