## 核心设计原则：休眠优先

分身的默认状态是**休眠**，不是在线。激活只发生在以下三种情况：

1. **调度器定时唤醒**：由分身自己生成的唤醒计划触发，执行完毕后自动回到休眠。
2. **用户显式呼叫**：用户在对话中 @分身，唤醒后处理完当前请求即进入休眠倒计时。
3. **事件触发唤醒**：绑定的关键事件（如尽调审批完成、HITL 通知）触发分身唤醒处理。

休眠期间零 Token 消耗，分身状态持久化在 PG + L3，唤醒时从 L2/L3 恢复上下文，不依赖进程内存。

### 任务拆分

#### T3.1 Persona 服务核心

**类型**：独立任务
**预估工时**：6天

**范围**：
- 实现 `PersonaProfile`：分身元数据（名称、能力标签、授权范围、绑定用户、创建时间）
- 实现 `PersonaState`：完整分身状态机（见下方状态机定义），持久化 PG
- 实现 Persona CRUD API：创建/查询/更新/停用，含 Harness 工具白名单注册
- 实现分身停用时的在途任务处理策略（见下方补充）
- 实现分身记忆主动删除 API（合规遗忘权）
- PG 表：`persona_profiles`、`persona_states`（含 RLS，按 `owner_id` 隔离）

**完整分身状态机（必须在 T3.1 交付前确认，PG 枚举与代码枚举须保持一致）**：

```
sleeping ──(用户呼叫 / 调度触发 / 事件触发)──▶ waking
                                                   │
                                          上下文恢复成功
                                                   │
                                                   ▼
                                               active ──(idle_timeout)──▶ sleeping
                                                   │
                                          (代理执行 HITL 等待)
                                                   │
                                                   ▼
                                         waiting_approval
                                          │             │
                                      owner 批准     owner 拒绝
                                          │             │
                                          ▼             ▼
                                        active        sleeping
                                          
sleeping / waking / active / waiting_approval
          ──(owner 停用)──▶ suspended
          
waking ──(上下文恢复失败)──▶ error ──(人工处理后重置)──▶ sleeping
```

所有状态的持久化字段：`{ persona_id, state, entered_at, previous_state, trigger_type, error_reason }`。

**状态转移条件与触发者说明**：

| 转移 | 触发者 | 条件 |
|------|-------|------|
| sleeping → waking | 用户呼叫 / 调度器 / 事件总线 | 任一唤醒信号到达（需先抢幂等唤醒锁，见 T3.2） |
| waking → active | Persona 服务 | L2/L3 上下文恢复成功 |
| waking → error | Persona 服务 | 上下文恢复失败，写 HUMAN_REVIEW 告警 |
| active → waiting_approval | PersonaExecutionGuard | 代理执行遇到 HIGH_WRITE 操作 |
| active → sleeping | Persona 服务 | idle_timeout 计时到期 |
| waiting_approval → active | HITLApprovalStateMachine | owner 批准 |
| waiting_approval → sleeping | HITLApprovalStateMachine | owner 拒绝 / HITL 超时否决 |
| * → suspended | owner 显式停用 | 见停用处理策略 |

**分身停用时的在途任务处理策略**：

owner 停用分身时，Persona 服务必须按以下顺序处理：

1. 立即暂停调度器中该 persona 的所有 `pending` 任务（状态改为 `paused`，不删除，保留可恢复记录）
2. 已进入 `waking` / `active` / `waiting_approval` 状态的任务，允许完成当前执行周期后再切换到 `suspended`，不强制中断
3. 停用 30 天后触发归档流程时，对应调度任务永久取消，写入 `AuditStoreFacade`（entity_type=persona, event_type=persona_archived）

**分身记忆主动删除 API（合规遗忘权）**：

按照个人信息保护法和金融行业合规要求，owner 有权主动删除自己分身的记忆数据。实现要求：

- API：`DELETE /api/v1/persona/{persona_id}/memory`，支持按时间范围过滤
- 删除操作从 Milvus Collection 中物理删除对应向量条目
- 删除内容不可查（不保留被删记忆原文），但删除行为必须写 WAL + `AuditStoreFacade` 留存（合规轨保留 7 年）
- 审计记录格式：`{ persona_id, owner_id, delete_range_start, delete_range_end, deleted_count, operator_id, ts }`
- 删除接口受 ABAC 控制，仅 owner 本人和具备 `persona:memory_admin` 权限的审计人员可调用

**细节补充**：
- 每个用户最多创建 N 个分身（N 由 Nacos 配置，默认 3），防止 Token 配额被滥用。`[待讨论，建议默认 3]`
- 分身停用后进入 `suspended` 态，历史记忆保留，不物理删除
- 分身能力标签决定可调用的工具集合，标签由 Harness 白名单配置文件映射，不允许用户自由填写。`[待讨论：能力标签具体定义和白名单映射规则，建议由平台架构师 + 合规团队共同维护]`
- 所有分身操作通过 `AuditStoreFacade` 写入 `entity_type=persona` 审计记录

**验收标准**：用户可创建分身并查询状态；停用分身后历史记忆可查询；跨用户访问被 RLS 拦截；停用时 pending 调度任务自动 paused；记忆删除留有审计记录

#### T3.2 休眠与唤醒机制

**类型**：依赖 T3.1（接口依赖——需要完整 PersonaState 状态机确定后实现）
**预估工时**：5天

**范围**：
- 实现休眠状态机：`active → sleeping`（无任务超过 idle_timeout 自动休眠）
- 实现三类唤醒触发：显式呼叫唤醒 / 调度器定时唤醒 / 事件触发唤醒
- 实现**唤醒幂等锁**：防止三类触发并发时多次唤醒（见下方设计）
- 实现唤醒时 L2/L3 上下文恢复：按恢复策略从 PG 投影 + Milvus 语义召回重建分身工作上下文
- 唤醒任务执行完毕后，自动启动休眠倒计时（idle_timeout，默认 5 分钟，Nacos 可配）

**唤醒幂等锁设计（必须实现，防止三路并发唤醒）**：

三类唤醒触发（用户呼叫、调度器触发、事件总线）可能在同一时刻到达，不加保护会导致：上下文被多次恢复覆盖、waking 状态被重复写入、Token 三倍消耗。

实现要求：
- 任一唤醒路径在将状态写入 PG 前，必须先用 Redis `SETNX` 抢占幂等锁：key 格式 `persona:wakeup_lock:{persona_id}`，TTL = `waking` 阶段最大允许耗时（建议 30s）
- 抢到锁的路径正常执行唤醒流程
- 未抢到锁的路径进入等待队列（不静默丢弃），等当前 `waking→active` 周期完成后，由 Persona 服务检查是否仍有排队唤醒需求，如有则合并为一次执行
- 若 Redis 不可用，降级为 PG `SELECT FOR UPDATE` 行级锁，同样保证幂等

**唤醒时 L2/L3 上下文恢复策略（明确恢复边界，防止 Token 超支）**：

上下文恢复本身计入 `simple` 预算（5,000 tokens），超出则裁剪旧内容。具体恢复层次：

| 层 | 恢复内容 | 边界 |
|----|---------|------|
| L1 | 当前活跃 Thread 最近 K 轮对话（默认 5 轮） | 仅恢复 `status=active` 或最近 24h 内有写入的 Thread |
| L2 | Thread 状态投影（`redis_projection` 或 PG 最近 checkpoint） | 仅恢复 `pending` 状态或最近 24h 内活跃的 Thread，不全量回放历史 Thread |
| L3 | 基于唤醒触发事件的 embedding 召回最相关记忆 | Top-K=5，score 阈值 ≥ 0.75，低于阈值不强制填充 |

恢复失败处理：L2/L3 任一恢复步骤抛出异常，状态切换为 `error`，写 `HUMAN_REVIEW` 告警，通知 owner，不得在恢复不完整的情况下直接响应用户。

**细节补充**：
- 休眠期间不持有任何 LLM 连接，不消耗 Token，仅保持 PG 状态记录
- 唤醒过程：`sleeping → waking`（上下文恢复中）→ `active`（就绪）
- 显式呼叫唤醒：通过 Semantic Router 新意图 `persona_invoke` 触发，携带 `persona_id`
- 事件触发唤醒：订阅 RocketMQ/ONS 的特定事件 topic（如 `hitl.approved`、`dd_plan.completed`），由分身在创建时配置订阅列表
- 唤醒后的上下文恢复过程必须写 WAL，恢复成功和失败均记录

**验收标准**：分身空闲超过 idle_timeout 自动进入 sleeping；用户 @分身后 P99 < 5s 恢复首条回复；休眠期间 Token 消耗为 0；三类触发并发时，有且仅有一次唤醒被执行

#### T3.3 分身记忆体系（Persona Collection 策略）

**类型**：依赖 T3.1（接口依赖——需要 persona_id 确定后创建 Collection）
**预估工时**：5天（含 Collection 策略决策）

**范围**：
- 按分身风险等级选择 Milvus Collection 策略（见下方策略定义）
- 实现分身记忆写入：用户交互摘要、执行任务结果、关键决策指纹写入 L3
- 实现分身记忆召回：唤醒时按相关性召回最近 K 条记忆，融合 L2 结构化状态
- 实现分身记忆主动删除（由 T3.1 提供 API，本任务提供 Milvus 层删除实现）
- 分身记忆 TTL：默认 180 天，用户可在授权范围内设置，超期归档到 OSS。`[待讨论：TTL 默认值和可调范围]`

**Milvus Collection 策略（解决 Collection 数量爆炸问题）**：

Milvus 单集群 Collection 数量存在实际运维上限（生产环境建议控制在 500 个以内），若按每用户 3 个分身计算，1000 个用户将产生 3000 个 Collection，严重超出推荐值。因此必须按分身敏感级别选择隔离策略：

| 分身类型 | Collection 策略 | 说明 |
|---------|----------------|------|
| 普通分身（无尽调/审批权限） | 共享 Collection + `persona_id` Partition | Collection 名：`persona_memory_shared_{tenant_id}` |
| 高敏分身（绑定尽调/审批/合规权限） | 独立 Collection | Collection 名：`persona_{persona_id}_memory` |

Collection 总数监控与治理：
- Milvus Collection 总数告警阈值：> 300 时触发运维告警
- 若普通分身单个 Partition 数据量占该 Collection 总量 < 10%，召回 recall 可能退化（参考一/二期 §4.4.3 ANN Recall 权衡），触发迁移评估
- 每季度执行 Collection 健康检查，对不活跃（180 天无写入）的分身 Collection/Partition 执行归档

分身 Collection 创建时机：高敏分身在 T3.1 创建 API 调用后同步创建；普通分身使用已有共享 Collection 中的 Partition，无需新建 Collection。

**细节补充**：
- 分身记忆不得包含其他用户的 PII 原文，写入前经 `PIITokenizationGateway` 掩码
- 分身被停用时，记忆 Collection/Partition 不立即删除，保留 30 天后归档（防止误停用）
- 记忆写入走 `AuditStoreFacade`，不直接操作 Milvus，方便后续接入记忆审计
- 主动删除 API 调用后，删除行为写合规审计轨，删除内容不可查

**验收标准**：不同分身的记忆严格隔离（Collection 或 Partition 级别均验证）；唤醒后上下文召回包含近期相关记忆；PII 字段已掩码；Collection 总数告警可触发；记忆删除审计记录可查

#### T3.4 Persona Skill 上架审核

**类型**：独立任务
**预估工时**：3天

**范围**：
- 实现 Persona 基础问答 Skill（`persona-qa-v1`）：基于分身记忆 + L2 上下文回答用户问题
- Skill 上架审核：红队扫描（≥ 10 个攻击向量，重点测试分身身份伪造和跨用户记忆访问）+ Evaluation 门禁
- Skill 元数据声明 `persona_id` 绑定，禁止跨分身调用

**细节补充**：
- 分身回答必须附上「以 {分身名} 的身份回答」的身份前缀，不得让用户误以为是系统官方回复
- 分身不得声称自己是真实人类，被追问时必须如实告知 AI 身份
- Skill 预算：`token_budget_complexity: simple`（5,000 tokens），分身问答保持轻量

**验收标准**：红队身份伪造测试全部被拦截；分身问答结果有身份前缀；跨分身调用被 Harness 拒绝

#### T4.2 摘要生成与 Reflection 校验

**类型**：依赖 T4.1（接口依赖——需要收集结果 URI 就绪）
**预估工时**：4天

**范围**：
- 实现 `PersonaSummarySkill`：基于收集结果生成结构化摘要（待办事项、风险提示、进展更新）
- 实现 `PersonaReflectionGate`：校验摘要完整性，失败时通知 owner
- 摘要通过 Reflection 校验后写入分身 L3 记忆，作为下一步规划的上下文
- 所有分身主动触发的对外输出（含摘要推送通知）必须携带「AI 分身生成」标记（不仅限于代理执行写操作）

**PersonaReflectionGate 实现规范**：

Reflection 校验必须明确实现层级，避免引入不必要的 LLM 消耗：

第一层（Tier A，规则引擎，零 Token）：结构化 Schema 校验，验证三类必填字段存在且内容非空：
- `time_range`：时间范围字段，值非空字符串且包含可识别的日期格式
- `data_sources`：数据来源字段，值为非空数组且至少有一个有效来源
- `key_items`：关键事项字段，值为非空数组且至少有一个条目

第二层（Tier C，LLM，按需触发）：仅在以下情况调用，消耗计入 `simple` 预算：`key_items` 数组内容语义异常（如所有条目均为空白字符或无意义重复内容）。

重试与失败处理：
- Schema 校验失败：直接进入失败流程，不重试（结构缺失不通过 LLM 补全，防止幻觉污染摘要）
- LLM 语义校验失败：最多 1 次重试；仍失败则写 `HUMAN_REVIEW`，通知 owner，不静默丢弃

**摘要推送的透明度标记**：

分身生成摘要后向 owner 推送通知时，通知内容必须携带「AI 分身生成」标记，由 `PersonaSummarySkill` 在输出层强制注入，不经过 Prompt，不可被覆盖。格式建议：消息体前缀 `[{分身名} AI 生成]`。

**细节补充**：
- 摘要不存原始数据，只存结构化摘要指纹，原始数据 Claim-Check URI 保留 24h 后自动清理
- 摘要生成 Token 预算：`simple`（5,000 tokens）；若摘要涉及尽调报告等长文档，升至 `moderate` 并记录

**验收标准**：摘要包含时间范围/数据来源/关键事项三类字段且字段内容有效；Reflection 校验失败时通知 owner；摘要推送通知有「AI 分身生成」标记；摘要写入 L3 后可被唤醒召回

### 任务拆分

#### T5.1 受限代理执行守卫

**类型**：独立任务
**预估工时**：6天

**范围**：
- 实现 `PersonaExecutionGuard`：分身代理执行前的三层校验（owner 授权列表 / Harness 白名单 / 风险等级评估）
- 实现操作风险分级：只读操作（`READ`）直接执行；低风险写操作（`LOW_WRITE`）执行后通知 owner；高风险写操作（`HIGH_WRITE`）强制转 HITL 等待 owner 确认
- 实现代理执行日志：每次代理执行写 WAL + `AuditStoreFacade`，记录操作类型、执行结果、是否需要 owner 确认
- 实现多步操作的事务原子性保障（见下方补充）

**高风险写操作清单（初始定义，由平台架构师 + 合规团队共同维护）**：`[待讨论：最终清单在三阶段启动前由平台架构师 + 合规团队共同确认]`

包括但不限于：提交审批决策、发送 IM 消息给外部联系人（非 owner 同一组织成员）、修改尽调 Plan 状态、触发任何涉及资产数量/金额变更的操作。

**代理执行事务原子性保障**：

分身代理执行某些操作可能涉及多个步骤（如：读取审批单 → 填写意见 → 提交），中途失败若无补偿，会产生审批单部分写入的危险中间态。

实现要求：

单步操作（一次 API 调用即可完成）：直接执行，失败写 WAL 告警，操作天然幂等。

多步操作（需要多次 API 调用组合完成）：必须在 Skill 元数据中声明为 Saga 序列，每步操作携带 `idempotency_key`，格式：`persona:{persona_id}:exec:{operation_type}:{target_id}:{date}`。任一步骤失败，触发已完成步骤的补偿回滚，补偿操作写 `AuditStoreFacade`（event_type=persona_exec_compensated）。不允许存在无补偿声明的多步写操作 Skill 上架。

**细节补充**：
- 分身**永远不能**代 owner 执行资金相关操作，这是继承自一期的绝对硬约束，由 Harness 层硬性拦截
- HITL 等待期间分身进入 `waiting_approval` 态，owner 批准后继续，拒绝后写告警并回休眠
- owner 可随时撤销分身的某类代理执行权限，撤销后分身在途任务不受影响，新任务立即生效

**验收标准**：高风险操作 100% 转 HITL；资金操作被 Harness 硬性拦截；代理执行日志可查；多步操作失败时补偿记录可查

| 收集失败率 | > 5% | 外部数据源不稳定信号 |
| 摘要 Reflection 通过率 | < 90% | 收集数据质量或 Skill 质量下降信号 |
| 唤醒计划注册成功率 | < 95% | 调度器异常或配额问题信号 |
| 代理执行 HITL 触发率 | > 30% | 高风险操作占比异常，可能存在误触或权限设置问题 |
| 分身月度 Token 消耗（entity 维度） | > 80% 月度配额 | 成本控制预警 |
| 分身 DLQ 积压 | > 10 | 调度器消费滞后信号 |
| 代理执行成功率 | < 95% | 代理执行质量下降信号 |
| 透明度标记注入失败次数 | > 0 | **任何一次失败即触发高优告警**，阈值为 0 |

**验收标准**：10 项告警全部配置并可触发；透明度标记失败计数告警阈值为 0，测试时故意触发一次注入失败，告警必须发出

#### T5.5 全链路压测 + 安全审计

**类型**：依赖 T5.1 ~ T5.4（降级依赖——建议全部三期服务就绪后进行）
**预估工时**：6天

**范围**：
- 并发唤醒压测：模拟 100 个活跃分身同时被调度器唤醒，P99 < 10s，Error Rate < 1%
- 三期红蓝对抗：分身身份伪造测试 + 代理执行越权测试 + 资金操作绕过测试 + 休眠绕过攻击测试 + 记忆跨用户访问测试 + 透明度标记绕过测试 + 并发唤醒幂等绕过测试（新增）
- 第三方安全审计：重点审查 `PersonaExecutionGuard` 三层校验、「AI 分身代发」标记注入、分身记忆 Collection/Partition 隔离、唤醒幂等锁有效性

**细节补充**：
- 压测数字（100 个并发唤醒）基于预期用户规模估算，实际以业务量确认后为准
- 对抗测试样本全部存储为可重放测试用例，纳入 CI 回归

**验收标准**：压测指标达标；红蓝对抗无高危漏洞；第三方审计报告输出；对抗样本纳入 CI

## 风险与应对

| 风险 | 影响阶段 | 应对措施 |
|------|----------|----------|
| 分身身份伪造绕过透明度标记 | 三阶段 | 标记由各输出层模块强制注入，不经过 Prompt，覆盖所有对外输出类型，红队测试 ≥ 15 个向量；透明度标记失败次数告警阈值为 0 |
| 三类唤醒触发并发导致多次唤醒 | 一/二/三阶段 | Redis SETNX 幂等锁 + 降级为 PG SELECT FOR UPDATE，并发唤醒压测纳入 T5.5 对抗场景 |

| 多步代理执行中途失败产生危险中间态 | 三阶段 | 多步操作必须声明 Saga 序列 + 幂等键；任一步失败触发补偿回滚；无 Saga 声明的 Skill 禁止上架 |
| 分身记忆跨用户泄露 | 一/二/三阶段 | 独立 Collection 或 Partition 策略 + RLS + Reflection 写入前 PII 掩码 + 二次校验；高敏分身默认独立 Collection |

| 分身唤醒计划暴涨导致调度器过载 | 二/三阶段 | 单分身最多注册 M 个计划 + persona 月度配额限制 + 调度器 DLQ 告警 |
| 分身 Token 月度超限 | 二/三阶段 | persona 级配额独立计算，80% 告警，超限写 HUMAN_REVIEW 通知 owner（不自动重试） |
| 分身被停用后在途任务继续执行 | 一阶段 | 停用触发时立即 pause 调度器 pending 任务；已进入执行周期的任务允许完成当前周期；30 天归档时永久取消并写审计 |
| 分身记忆无法主动删除导致合规风险 | 一/三阶段 | T3.1 提供记忆主动删除 API；删除内容不保留但删除行为写合规审计轨（保留 7 年） |

| 二期调度器不稳定导致唤醒失败 | 二阶段 | 三期启动前置门禁要求调度器稳定运行 ≥ 30 天 |
| owner 误授权导致分身权限过大 | 三阶段 | 授权列表由 Harness 白名单约束，用户不能超出平台允许范围自行扩大权限 |
