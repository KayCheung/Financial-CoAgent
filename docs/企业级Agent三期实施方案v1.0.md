## 三期启动前提

本方案基于二期已全面投产（二期 M8 完成）。三期开发不得在以下门禁未满足时启动：

| 门禁项 | 验收标准 | 检查方 |
|--------|----------|--------|
| 二期调度器稳定运行 ≥ 30 天 | cron 任务成功率 ≥ 95%，DLQ 积压 < 10 | 安全/运维 |
| 二期 L3 语义记忆稳定运行 ≥ 30 天 | recall@5 ≥ 0.85，跨租户零泄露 | AI 算法 |
| 二期 AuditStoreFacade 稳定运行 ≥ 30 天 | WAL 未同步条目数长期 < 10 | 安全/运维 |
| Temporal 调度器评估报告已输出 | M8 评估报告存在，三期调度器选型已定 | 平台架构师 |
| 二期红蓝对抗演练通过 | 二期新增功能无高危漏洞 | 安全/运维 |

## 核心设计原则：休眠优先

分身的默认状态是**休眠**，不是在线。激活只发生在以下三种情况：

1. **调度器定时唤醒**：由分身自己生成的唤醒计划触发，执行完毕后自动回到休眠。
2. **用户显式呼叫**：用户在对话中 @分身，唤醒后处理完当前请求即进入休眠倒计时。
3. **事件触发唤醒**：绑定的关键事件（如尽调审批完成、HITL 通知）触发分身唤醒处理。

休眠期间零 Token 消耗，分身状态持久化在 PG + L3，唤醒时从 L2/L3 恢复上下文，不依赖进程内存。

## 第一阶段：分身基座（P3-M1-M2，8周）

### 阶段目标

**让用户能创建并感知到自己的 AI 分身**：分身可被创建、配置、休眠、唤醒，能基于用户历史记忆回答基础问题。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 分身创建 | 用户可创建分身，配置名称/能力标签/授权范围，10s 内完成初始化 |
| 休眠机制 | 分身无任务时自动进入休眠，休眠期间 Token 消耗为 0 |
| 唤醒延迟 | 用户显式呼叫后，分身从休眠恢复并返回首条回复 P99 < 5s |
| 记忆隔离 | 不同用户的分身记忆严格隔离，跨用户访问被 RLS 拦截 |
| Evaluation 基线 | 分身问答样例集 ≥ 30 条，基线建立完成 |

### 新增/升级中间件

| 中间件 | 用途 | 部署要求 | 最低资源 |
|--------|------|----------|----------|
| Persona 服务（新增） | 分身 CRUD + 状态管理 + 休眠/唤醒调度 | 2 副本 | 2C2G × 2 |
| Milvus（二期已有） | 新增 persona 专属 Collection | 复用二期 | 按用量计费 |
| PG（一/二期已有） | persona 元数据表 + 状态表 | 复用一/二期 | 0 |

> **阶段一资源增量**：约 +4C +4G

### 任务拆分

#### T3.1 Persona 服务核心

**类型**：独立任务
**预估工时**：6天

**范围**：
- 实现 `PersonaProfile`：分身元数据（名称、能力标签、授权范围、绑定用户、创建时间）
- 实现 `PersonaState`：分身运行态（sleeping / waking / active / suspended），持久化 PG
- 实现 Persona CRUD API：创建/查询/更新/停用，含 Harness 工具白名单注册
- PG 表：`persona_profiles`、`persona_states`（含 RLS，按 `owner_id` 隔离）

**细节补充**：
- 每个用户最多创建 N 个分身（N 由 Nacos 配置，默认 3），防止 Token 配额被滥用
- 分身停用后进入 `suspended` 态，历史记忆保留，不物理删除
- 分身能力标签决定可调用的工具集合，标签由 Harness 白名单配置文件映射，不允许用户自由填写
- 所有分身操作通过 `AuditStoreFacade` 写入 `entity_type=persona` 审计记录

**验收标准**：用户可创建分身并查询状态；停用分身后历史记忆可查询；跨用户访问被 RLS 拦截

---

#### T3.2 休眠与唤醒机制

**类型**：依赖 T3.1（接口依赖——需要 PersonaState 模型确定）
**预估工时**：5天

**范围**：
- 实现休眠状态机：`active → sleeping`（无任务超过 idle_timeout 自动休眠）
- 实现三类唤醒触发：显式呼叫唤醒 / 调度器定时唤醒 / 事件触发唤醒
- 实现唤醒时 L2/L3 上下文恢复：从 PG 投影 + Milvus 语义召回重建分身工作上下文
- 唤醒任务执行完毕后，自动启动休眠倒计时（idle_timeout，默认 5 分钟，Nacos 可配）

**细节补充**：
- 休眠期间不持有任何 LLM 连接，不消耗 Token，仅保持 PG 状态记录
- 唤醒过程：`sleeping → waking`（上下文恢复中）→ `active`（就绪），恢复失败则写 `HUMAN_REVIEW` 告警
- 显式呼叫唤醒：通过 Semantic Router 新意图 `persona_invoke` 触发，携带 `persona_id`
- 事件触发唤醒：订阅 RocketMQ/ONS 的特定事件 topic（如 `hitl.approved`、`dd_plan.completed`），由分身在创建时配置订阅列表
- 唤醒后的上下文恢复必须写 WAL，恢复失败不得直接响应用户，必须先报错

**验收标准**：分身空闲超过 idle_timeout 自动进入 sleeping；用户 @分身后 P99 < 5s 恢复首条回复；休眠期间 Token 消耗为 0

---

#### T3.3 分身记忆体系（Persona L3 Collection）

**类型**：依赖 T3.1（接口依赖——需要 persona_id 确定后创建隔离 Collection）
**预估工时**：4天

**范围**：
- 为每个分身创建独立 Milvus Collection（`persona_{persona_id}_memory`），不与其他用户/分身共享
- 实现分身记忆写入：用户交互摘要、执行任务结果、关键决策指纹写入 L3
- 实现分身记忆召回：唤醒时按相关性召回最近 K 条记忆，融合 L2 结构化状态
- 分身记忆 TTL：默认 180 天，用户可在授权范围内设置，超期归档到 OSS

**细节补充**：
- 分身记忆不得包含其他用户的 PII 原文，写入前经 `PIITokenizationGateway` 掩码
- 分身被停用时，记忆 Collection 不立即删除，保留 30 天后归档（防止误停用）
- 记忆写入走 `AuditStoreFacade`，不直接操作 Milvus，方便三期后接入记忆审计

**验收标准**：不同分身的记忆 Collection 完全隔离；唤醒后上下文召回包含近期相关记忆；PII 字段已掩码

---

#### T3.4 Persona Skill 上架审核

**类型**：独立任务
**预估工时**：3天

**范围**：
- 实现 Persona 基础问答 Skill（`persona-qa-v1`）：基于分身记忆 + L2 上下文回答用户问题
- Skill 上架审核：红队扫描（≥ 10 个攻击向量，重点测试分身身份伪造和跨用户记忆访问）+ Evaluation 门禁
- Skill 元数据声明 `persona_id` 绑定，禁止跨分身调用

**细节补充**：
- 分身回答必须附上"以 {分身名} 的身份回答"的身份前缀，不得让用户误以为是系统官方回复
- 分身不得声称自己是真实人类，被追问时必须如实告知 AI 身份
- Skill 预算：`token_budget_complexity: simple`（5,000 tokens），分身问答保持轻量

**验收标准**：红队身份伪造测试全部被拦截；分身问答结果有身份前缀；跨分身调用被 Harness 拒绝

## 第二阶段：自主收集与规划（P3-M3-M4，8周）

### 阶段目标

**让分身具备主动感知能力**：分身可在授权范围内自动收集信息、生成摘要、分析优先级，并自主生成下一步行动计划注册为定时任务。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 信息收集 | 分身可从授权数据源（日历/看板/尽调/审批）自动收集信息，单次收集 Token 消耗 ≤ simple 预算 |
| 摘要质量 | 摘要生成后经 Reflection 校验，关键字段非空率 ≥ 95% |
| 唤醒计划生成 | 分身可基于摘要生成结构化唤醒计划，并自动注册到调度器 |
| 调度准确性 | 分身注册的定时任务按计划触发，成功率 ≥ 95% |
| Token 成本 | 单次自主收集 + 摘要 + 规划全链路 Token 消耗 ≤ 10,000（moderate 预算以内） |

### 新增中间件

| 中间件 | 用途 | 部署要求 | 最低资源 |
|--------|------|----------|----------|
| 无新增（复用二期调度器和 RocketMQ/ONS） | — | — | 0 |

> **阶段二资源增量**：约 +0C +0G（纯逻辑扩展）

### 任务拆分

#### T4.1 分身信息收集工具集

**类型**：独立任务
**预估工时**：5天

**范围**：
- 实现授权数据源收集 Tool 集合：`persona_calendar_fetch`、`persona_kanban_fetch`、`persona_dd_plan_fetch`、`persona_approval_fetch`
- 所有收集工具严格受 Harness 白名单约束，收集范围限定在分身 `owner_id` 授权数据
- 收集结果写 Claim-Check（OSS URI），不将原始数据写入 `AgentFastState`
- 收集频率限制：单个分身每小时最多触发 N 次收集（Nacos 配置，默认 4 次/小时）

**细节补充**：
- 收集工具的 `scope` 严格对齐二期 `source_access_policy`，不因分身身份扩大数据可见性
- 管理员分身不自动获得更大数据范围，需显式授予 `delegated_view` 范围
- 收集失败（数据源不可用）映射 `RETRYABLE`，连续 3 次失败后跳过本次收集并写告警

**验收标准**：收集结果仅包含 owner 授权数据；超出授权范围的数据源访问被 Harness 拒绝；收集频率限制生效

---

#### T4.2 摘要生成与 Reflection 校验

**类型**：依赖 T4.1（接口依赖——需要收集结果 URI 就绪）
**预估工时**：4天

**范围**：
- 实现 `PersonaSummarySkill`：基于收集结果生成结构化摘要（待办事项、风险提示、进展更新）
- 实现 `PersonaReflectionGate`：校验摘要完整性（至少包含时间范围、数据来源、关键事项三类字段）
- 摘要通过 Reflection 校验后写入分身 L3 记忆，作为下一步规划的上下文

**细节补充**：
- 摘要不存原始数据，只存结构化摘要指纹，原始数据 Claim-Check URI 保留 24h 后自动清理
- Reflection 校验失败 → `HUMAN_REVIEW`，通知分身 owner，不静默丢弃
- 摘要生成 Token 预算：`simple`（5,000 tokens）；若摘要涉及尽调报告等长文档，升至 `moderate` 并记录

**验收标准**：摘要包含时间范围/数据来源/关键事项三类字段；Reflection 校验失败时通知 owner；摘要写入 L3 后可被唤醒召回

---

#### T4.3 唤醒计划生成与调度注册

**类型**：依赖 T4.2（接口依赖——需要摘要就绪后生成规划）
**预估工时**：5天

**范围**：
- 实现 `PersonaPlannerSkill`：基于摘要分析优先级，生成结构化唤醒计划（任务列表 + 建议唤醒时间 + 预估 Token 消耗）
- 实现计划到调度器的自动注册：将唤醒计划转换为 `ScheduleJob`（one_shot 或 cron），通过 `WorkflowTriggerGateway` 注册
- 分身每次注册计划前，先检查调度器中是否已有未执行的同类计划，避免重复注册

**细节补充**：
- 分身单次最多注册 M 个唤醒任务（M 由 Nacos 配置，默认 5），防止调度器被单个分身占满
- 唤醒计划必须包含预估 Token 消耗，超出 persona 月度配额时，分身自动降级或跳过低优先级任务
- 计划生成本身的 Token 消耗计入 `entity_type=persona` 的月度 Token 聚合视图
- 所有注册操作通过 `AuditStoreFacade` 写审计，分身 owner 可查看分身注册了哪些计划

**验收标准**：分身生成的唤醒计划成功注册到调度器；重复计划被去重；月度 Token 配额超限时自动降级

---

#### T4.4 Persona Token 配额扩展

**类型**：依赖 T3.1（接口依赖——需要 persona_id 维度）
**预估工时**：3天

**范围**：
- 扩展 `TokenBudget`，新增 `entity_type=persona` 维度聚合
- Nacos 配置 persona 级月度 Token 上限（默认 500,000 tokens/月/分身），告警阈值 80%
- 实现分身配额视图：owner 可查看分身本月 Token 消耗明细（按唤醒任务类型分类）

**细节补充**：
- 分身 Token 配额独立于用户的对话配额，两者分开计算和告警
- 超出月度配额时，分身自动进入降级模式：只执行高优先级唤醒任务，跳过低优先级
- 降级模式下每次唤醒前检查剩余配额，不足则跳过并写告警

**验收标准**：persona Token 消耗可按月聚合查询；超限时自动降级；owner 可看到消耗明细

## 第三阶段：代理执行与生产就绪（P3-M5-M8，16周）

### 阶段目标

**让分身具备受限代理执行能力，并全面投产**：分身可在 owner 明确授权的范围内代为执行有限任务，同时完成全链路观测、压测和安全审计。

### 阶段完成指标

| 指标 | 达成标准 |
|------|----------|
| 代理执行 | 分身可代 owner 执行授权范围内任务（如回复评论、提交审批意见），高风险操作 100% 转 HITL |
| 执行透明度 | 所有代理执行有操作日志，owner 可随时查看分身代为执行的历史 |
| 压测通过 | 100 个活跃分身并发唤醒场景 P99 < 10s，Error Rate < 1% |
| 安全审计通过 | 三期新增功能通过红蓝对抗 + 第三方安全审计 |
| Token 成本 | 100 个分身全月 Token 消耗在预算范围内，无超限熔断 |

### 新增中间件

| 中间件 | 用途 | 部署要求 | 最低资源 |
|--------|------|----------|----------|
| 无新增（复用一/二期全部中间件） | — | — | 0 |

> **阶段三资源增量**：约 +0C +0G（纯逻辑扩展，调度器/Milvus 按用量弹性扩容）

### 任务拆分

#### T5.1 受限代理执行守卫

**类型**：独立任务
**预估工时**：6天

**范围**：
- 实现 `PersonaExecutionGuard`：分身代理执行前的三层校验（owner 授权列表 / Harness 白名单 / 风险等级评估）
- 实现操作风险分级：只读操作（`READ`）直接执行；低风险写操作（`LOW_WRITE`）执行后通知 owner；高风险写操作（`HIGH_WRITE`）强制转 HITL 等待 owner 确认
- 实现代理执行日志：每次代理执行写 WAL + `AuditStoreFacade`，记录操作类型、执行结果、是否需要 owner 确认

**细节补充**：
- 高风险写操作包括但不限于：提交审批决策、发送 IM 消息给外部联系人、修改尽调 Plan 状态
- 分身**永远不能**代 owner 执行资金相关操作，这是继承自一期的绝对硬约束
- HITL 等待期间分身进入 `waiting_approval` 态，owner 批准后继续，拒绝后写告警并回休眠
- owner 可随时撤销分身的某类代理执行权限，撤销后分身在途任务不受影响，新任务立即生效

**验收标准**：高风险操作 100% 转 HITL；资金操作被 Harness 硬性拦截；代理执行日志可查

---

#### T5.2 分身身份透明度机制

**类型**：依赖 T5.1（接口依赖——需要代理执行框架就绪）
**预估工时**：3天

**范围**：
- 所有分身对外输出（IM 消息、评论、审批意见）必须携带「AI 分身代发」标记
- 实现分身操作溯源：接收方可查询某条消息是否由 AI 分身代发，以及 owner 是谁
- 禁止分身在任何情况下声称自己是真实人类；被追问时必须如实告知

**细节补充**：
- 「AI 分身代发」标记不可被 Prompt 覆盖，由 `PersonaExecutionGuard` 在输出层强制注入
- 溯源查询接口受 ABAC 控制，只有 owner 本人、管理员、合规审计人员可访问

**验收标准**：所有代发内容有「AI 分身代发」标记；红队尝试覆盖标记的测试全部被拦截；溯源查询权限正确隔离

---

#### T5.3 Persona 代理执行 Skill 上架

**类型**：依赖 T5.1 + T5.2（接口依赖——需要守卫和透明度机制就绪后评测）
**预估工时**：4天

**范围**：
- 实现代理执行 Skill（`persona-exec-v1`）：封装低风险写操作执行能力
- Skill 上架审核：红队扫描重点测试身份伪造、权限越界、资金操作绕过（≥ 15 个攻击向量）+ Evaluation 门禁
- 初始灰度：canary 5%，仅向内部测试用户开放

**细节补充**：
- 相比二期，三期红队攻击向量数量提升到 ≥ 15 个，重点覆盖分身身份伪造场景
- 代理执行 Skill 只允许执行白名单内的操作类型，不允许调用任意工具

**验收标准**：15 个攻击向量全部被拦截；Evaluation 门禁通过；灰度 5% 流量稳定运行 7 天后扩灰

---

#### T5.4 三期可观测性扩展

**类型**：独立任务
**预估工时**：4天

**范围**：
- 新增 OTel Span：`persona.wakeup`、`persona.collect`、`persona.plan`、`persona.exec`
- 新增 10 项告警指标：分身唤醒延迟 P99、分身休眠成功率、收集失败率、摘要 Reflection 通过率、唤醒计划注册成功率、代理执行 HITL 触发率、分身月度 Token 消耗、分身 DLQ 积压、代理执行成功率、分身透明度标记覆盖失败次数（应为 0）
- Grafana 新增三期 Dashboard：分身活跃度热图、Token 消耗趋势、代理执行漏斗视图

**验收标准**：10 项告警全部配置并可触发；透明度标记覆盖失败次数告警阈值为 0（任何一次失败即告警）

---

#### T5.5 全链路压测 + 安全审计

**类型**：依赖 T5.1 ~ T5.4（降级依赖——建议全部三期服务就绪后进行）
**预估工时**：6天

**范围**：
- 并发唤醒压测：模拟 100 个活跃分身同时被调度器唤醒，P99 < 10s，Error Rate < 1%
- 三期红蓝对抗：分身身份伪造测试 + 代理执行越权测试 + 资金操作绕过测试 + 休眠绕过攻击测试 + 记忆跨用户访问测试
- 第三方安全审计：重点审查 `PersonaExecutionGuard` 三层校验、「AI 分身代发」标记注入、分身记忆 Collection 隔离

**细节补充**：
- 压测数字（100 个并发唤醒）基于预期用户规模估算，实际以业务量确认后为准
- 对抗测试样本全部存储为可重放测试用例，纳入 CI 回归

**验收标准**：压测指标达标；红蓝对抗无高危漏洞；第三方审计报告输出；对抗样本纳入 CI
